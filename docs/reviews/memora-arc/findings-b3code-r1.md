## Rejected / no-change (with reasons)

- **Grok #1 (claimed BLOCKER: Me-floor refuses benign amends on User rows below personal) + Grok #5 (Agent floor vs public rows) — REJECTED as blockers.** Two grounds. (1) Spec-mandated: the ratified amendment text reads "A classification that needs a tier higher than the stored memory's sensitivity/storage posture returns `MetadataAmendmentTierIncreaseRefused`; the operation does not re-tier" — refusing rather than uplifting is the contract, and the namespace floor is part of classification by design (June Me-floor lesson). (2) Empirically empty: full census of the live store (929 rows) shows user=29 all personal+encrypted, agent=3 all internal, project=897 all internal — zero rows where the floor exceeds stored tier, so the refusal cannot fire on the W5 backfill target. Hypothetical legacy violators surface as visible `tier_increase_refused` skips in the compile report — conservative and auditable, not silent. B3C-3 pins this behavior in tests.
- **Grok #3 / Opus #4 (encrypted amend → event append crash window) — NO CHANGE.** Sided with Opus: same write-then-event pattern used across Stream A; event-append *failure* is repaired via the durable enqueue; a hard crash between commit and append loses only the event, and the spec names git history as the primary audit trail. Opus verified the `committed:true, indexed:true` WriteOutcome is only surfaced on failure returns, where it is accurate (the canonical write did commit). Recorded as a known property in the design note's risk register.
- **Grok #6 (ROI dashboard blind to MetadataAmended) — deferred, out of B3 scope.** Observability follow-up noted for arc closeout; amend volume is visible in the event log and compile report.
- **Opus #3 (operational errors abort the whole backfill run) — NO CHANGE.** Fail-loud on infra faults is deliberate; CLI §8's outcome set is closed. W5 rehearsal runbook notes the operator expectation.
