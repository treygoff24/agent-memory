# Codex verification — Stream E Ambient Recall v1.1 patch (B8, B4)

**Date:** 2026-06-22. **Lane:** `delegate codex work` (OpenAI Codex, real-workspace read-only verification). **Purpose:** ground-truth the two factual claims that gated the v1.0 → v1.1 spec patch before incorporating them. Companion to the fusion review at `docs/reviews/2026-06-22-ambient-recall-v1-fusion.md`.

## What was asked

1. **B8 (prompt cache):** Do Claude Code / Codex inject per-turn `UserPromptSubmit` (and `SessionStart`, `PreCompact`) hook context into the **cached prompt prefix** (which would bust prefix caching every turn), or at the **bottom of context** adjacent to the latest user prompt (cache-safe)?
2. **B4 (daemon infra):** Does the `memoryd` daemon already have a background-worker / file-watch / cached-projection pattern the desk read could reuse, or is it net-new?

## How it changed the patch

- **B8 confirmed as a documentary contract, not a live defect.** Both Claude Code and Codex place `UserPromptSubmit` hook context in the *conversation* (turn-local), not the static system prompt; the Claude SDK explicitly notes conversation-injected content "doesn't affect the system prompt cache." So the spec mandates the *outcome* (dynamic recall stays outside the cached prefix) plus a Phase-1 placement assertion — not new machinery. (§9.5 updated to "confirmed.")
- **Factual correction — PreCompact is not a context-injection hook.** In **both** harnesses `PreCompact` is block-only (exit 2 / `decision:block` / `continue:false`) with **no `additionalContext` surface**. v1.0's T2 design fired PreCompact to *inject* re-orientation, which is impossible. v1.1 re-orients via **`SessionStart` source `compact`/`resume`** (already in the existing matcher) — so T2 needs **no new hook wire**. (§1.2, §3 intro, §3.3, §11, §14.1 updated.)
- **B4 grounded.** The daemon has reusable background-task idioms (reality-check `interval_at` scheduler, embedding worker shutdown-aware loop, `Substrate::watch()` file-watch via `notify`), but **no** git/desk projection cache — that `DeskProjection` state is net-new. (§8.0, §11 cite the concrete reuse targets.)

## Codex output (verbatim)

## Q1 — prompt-cache interaction with hook injection position

**VERDICT:** Current Claude Code and Codex do **not** put per-turn `UserPromptSubmit` hook context into the static top-level system prompt. Claude Code documents it as conversation-positioned "alongside the submitted prompt"; Codex docs call it "extra developer context," and official Codex source records it as conversation items after the submitted user input. I found no official sentence saying "hook output does/does not bust prompt cache," but the documented/source placement means B8 is mostly a rendering/contract requirement, not a current Claude Code defect.

**Evidence:**

- **Claude Code `UserPromptSubmit`: bottom/turn-local, not static system prompt.** Claude Code's hook docs say `additionalContext` is wrapped in a system reminder and "inserted into the conversation at the point where the hook fired"; event placement says `UserPromptSubmit` and `UserPromptExpansion` appear "alongside the submitted prompt." Source: Claude Code Hooks docs, `Add context for Claude`, lines 774-789 (`https://code.claude.com/docs/en/hooks`).
- **Claude Code `SessionStart`: start of conversation before first prompt.** Same placement table says `SessionStart`, `Setup`, and `SubagentStart` appear "at the start of the conversation, before the first prompt." Source: Claude Code Hooks docs, lines 784-786. SessionStart also reruns on resume rather than replaying stale mid-session hook text: Claude Code Hooks docs, lines 794-795.
- **Claude Code `PreCompact`: no context injection surface verified.** PreCompact is documented as a pre-compaction event that can block with exit code 2 or JSON `decision: "block"`; its input has `trigger` and `custom_instructions`, with no `additionalContext` output shape. Source: Claude Code Hooks docs, lines 2159-2180. The decision table lists PreCompact under top-level block decisions, while `SessionStart` is the context-only event for `additionalContext`. Source: Claude Code Hooks docs, lines 800-811.
- **Claude Code system-prompt/cache relationship:** the SDK docs distinguish system prompt mutation from conversation context: `CLAUDE.md`, skills, hooks, and permissions shape behavior "outside the system prompt"; `CLAUDE.md` is injected into the conversation, not the system prompt. Source: Claude Code SDK "Modifying system prompts," lines 142-146. The same doc says `CLAUDE.md` "doesn't affect the system prompt cache" because it is conversation-injected, not system-prompt-injected. Source: lines 241-244.
- **Claude prompt caching mechanics:** Claude prompt caching resumes from prompt prefixes; cache hits require exact matching of the prefix up to the breakpoint, and varying per-request suffixes should sit after the stable cached prefix. Sources: Claude Prompt Caching docs, lines 164-172, 301-318, 400-418, 612-627 (`https://platform.claude.com/docs/en/build-with-claude/prompt-caching`).
- **Codex `SessionStart`: extra developer context, before first input in source.** OpenAI Codex docs say `SessionStart` `stdout`/`additionalContext` is added as extra developer context, with source values `startup`, `resume`, `clear`, or `compact`. Source: OpenAI Codex Hooks docs, lines 935-954 (`https://developers.openai.com/codex/hooks`). Official Codex source runs pending SessionStart hooks before recording user input (`codex-rs/core/src/session/turn.rs:176-180`) and records their additional contexts as conversation items (`codex-rs/core/src/hook_runtime.rs:103-149`, `595-615`).
- **Codex `UserPromptSubmit`: turn-local developer context after submitted input in source.** OpenAI Codex docs say UserPromptSubmit plain stdout and `hookSpecificOutput.additionalContext` are added as extra developer context. Source: OpenAI Codex Hooks docs, lines 1129-1149. Official Codex source builds/runs the `UserPromptSubmitRequest` from the current user input (`codex-rs/core/src/hook_runtime.rs:500-527`), then for accepted input records the user prompt first and the hook additional contexts after it (`codex-rs/core/src/session/turn.rs:492-518`; `codex-rs/core/src/hook_runtime.rs:539-564`, `595-615`).
- **Codex `PreCompact`: no context injection surface verified.** Codex docs say PreCompact runs before compaction; plain stdout is ignored, JSON supports only Common output fields, and `continue:false` stops before compacting. Source: OpenAI Codex Hooks docs, lines 1105-1117; Common output fields are `continue`, `stopReason`, `systemMessage`, `suppressOutput`, lines 916-928.
- **OpenAI prompt caching mechanics:** OpenAI says cache hits are only possible for exact prefix matches, static content should be at the beginning, variable content at the end, and cache lookup checks whether the initial prompt prefix exists in cache. Source: OpenAI Prompt Caching docs, lines 717-735 (`https://developers.openai.com/api/docs/guides/prompt-caching`).

**IMPLICATION FOR THE SPEC:** Document a hard placement invariant: cue-driven per-turn recall must render as turn-local/tail conversation context adjacent to the latest user prompt, never by mutating the initial system/developer prefix. For Claude/Codex specifically, do not describe `PreCompact` as a context-injection hook; use the harness-supported post-compact/session-start path (`SessionStart` source `compact` where available) for re-orientation.

## Q2 — existing background-projection infrastructure in the daemon

**VERDICT:** The daemon already has long-lived background-task patterns and a substrate file-watch API, so a desk projection can reuse/sit beside existing infrastructure. But there is no existing daemon-level in-memory git/desk projection cache; that domain state and refresh logic would be net-new.

**Evidence:**

- **Background task pattern exists in `memoryd`.** `serve_substrate_with` wraps the substrate/state in `Arc`s and starts background components before serving: notification dispatcher, coordination cleanup, reality-check scheduler, and embedding worker (`crates/memoryd/src/server.rs:61-75`). The notification dispatcher is spawned with `tokio::spawn(dispatcher.run(receiver))` (`crates/memoryd/src/server.rs:85-89`) and consumes a broadcast loop (`crates/memoryd/src/notifications/dispatcher.rs:37-47`).
- **Periodic scheduler pattern exists.** `spawn_reality_check_scheduler` starts a `tokio::spawn` task with an hourly `interval_at`, skips missed ticks, and exits on shutdown (`crates/memoryd/src/server.rs:227-244`).
- **Embedding worker pattern exists.** The server's embedding worker loader intentionally avoids gating startup, sleeps briefly, loads the provider on `spawn_blocking`, retries with backoff, publishes the provider slot, then starts the drain worker (`crates/memoryd/src/server.rs:114-217`). The actual drain worker returns immediately from `spawn_embedding_worker` and runs a shutdown-aware loop draining `pending_embedding_jobs` (`crates/memoryd/src/embedding/worker.rs:1-7`, `59-67`, `96-152`).
- **File-watch pattern exists in `memory-substrate`.** `Substrate::watch()` exposes a watch subscription over the repo (`crates/memory-substrate/src/api/mod.rs:131-134`). The implementation uses `notify::recommended_watcher`, maps overflow to rescan-required events, filters paths, suppresses self-events, and watches the root recursively (`crates/memory-substrate/src/watcher/subscription.rs:69-118`).
- **Project binding git reads are exactly `rev-parse` then `remote get-url origin` under a 2s deadline.** `resolve_project_binding` first checks `.memory-project.yaml`; otherwise it calls `git_origin_remote` and hashes the normalized remote (`crates/memoryd/src/recall/project.rs:28-44`). `git_origin_remote` calls `git_worktree_root`, then `git remote get-url origin` (`crates/memoryd/src/recall/project.rs:145-153`). `git_worktree_root` calls `git rev-parse --show-toplevel` (`crates/memoryd/src/recall/project.rs:156-163`). `GIT_COMMAND_TIMEOUT` is 2 seconds, and `git_output` runs the command via `spawn_blocking` plus a timeout/poll loop (`crates/memoryd/src/recall/project.rs:15-18`, `165-188`).
- **Scope note on other git subprocesses:** project binding is not the only git subprocess use in the repo: e.g. dream lease code checks dirty user work with `git status --porcelain=v1 --untracked-files=all` (`crates/memoryd/src/dream/git.rs:168-176`), and substrate commit helpers run git add/diff/commit/rev-parse (`crates/memory-substrate/src/git/commit.rs:80-130`; `crates/memory-substrate/src/git/command.rs:23-48`). I did not find those on the Stream E recall hook path.
- **Existing cache/projection layers are not a desk projection.** `HandlerState` already owns in-memory registries/queues and an embedding-provider slot (`crates/memoryd/src/handlers/mod.rs:107-124`, `137-152`, `155-171`); `PassiveQueue` is an in-memory bounded queue (`crates/memoryd/src/notifications/passive.rs:18-55`); and substrate has a derived recall index / safe projection model (`crates/memory-substrate/src/model.rs:921-940`, `1357-1400`) plus embedding-job projection (`crates/memory-substrate/src/model.rs:1273-1281`). None of these stores branch/status/log/PR/CI desk state.

**IMPLICATION FOR THE SPEC:** Keep B4 as a real design requirement: desk/git state should be refreshed by a background daemon task and read O(1) from daemon state. The patch can reuse the existing `tokio::spawn` + shutdown + interval/watch patterns, but must add a new `DeskProjection`-style state object and refresh policy rather than doing synchronous `git status`/`git log`/`gh pr view` in SessionStart/PreCompact hooks.
