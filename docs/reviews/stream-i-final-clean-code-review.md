# Stream I Final Clean-Code Review

Status: Changes requested

## Findings

- Severity 2, `crates/memoryd/src/server.rs:53-90`, `crates/memoryd/src/handlers.rs:107-127`, `crates/memoryd/src/recall/delta.rs:120-124`, `crates/memoryd/src/recall/startup.rs:251-281`: Coordination configuration is defined and documented but not wired into daemon runtime. `serve_substrate` / `serve_substrate_with` always construct `HandlerState::new()`, which only uses hard-coded defaults; `HandlerState::with_coordination_level` only accepts a level and hard-codes claim-lock TTL and presence staleness; delta and startup recall construct `CoordinationConfig::default()` locally for the relevance gate. Why it matters: the spec requires a `config.yaml` `[coordination]` block with validation and fallback semantics, and docs say absent project mode falls back to daemon `coordination.level`. As implemented, non-default `threshold`, `recency_window_seconds`, `per_turn_cap`, `presence.stale_after_seconds`, and `claim_lock.ttl_seconds` cannot affect production behavior, and invalid coordination config cannot fail closed at daemon startup. This also leaves `memorum-coordination::CoordinationConfig` as a parallel/default-only model rather than the runtime source of truth. Suggested fix: add a validated coordination-config loader for repo `config.yaml`, construct `HandlerState` from the loaded `CoordinationConfig`, thread the configured relevance-gate values into `DeltaCoordinationContext` and startup peer-update construction, and add daemon-level tests that prove non-default config changes behavior plus invalid config refuses startup. Keep per-project `concurrent_session_mode` as an override on top of the loaded global level.
- Severity 3, `crates/memoryd/src/handlers.rs:91-105`, `crates/memoryd/src/handlers.rs:212-227`, `crates/memoryd/src/handlers.rs:404-452`, `crates/memoryd/src/recall/delta.rs:25-43`: Stream I daemon-local coordination state is partly ad hoc inside the already-large `handlers.rs` module. Presence and claim locks are cleanly delegated to `memorum-coordination`, but peer-delivery audit and peer-update cooldowns are private `StdMutex` containers in `handlers.rs`, surfaced through recall traits. Why it matters: the Stream I contract says coordination state logic belongs in the coordination crate and the review focus explicitly calls out no ad-hoc state. Keeping cooldown/audit storage in the request-handler module makes `handlers.rs` carry protocol dispatch, governance, privacy, web-dashboard process management, and Stream I state containers at once, which raises the cost of future concurrency or retention-policy changes and makes the cooldown/audit invariants harder to unit-test outside daemon integration tests. Suggested fix: move the bounded peer-delivery audit and per-receiver cooldown store into a small dedicated module, preferably `memorum-coordination` if it is generic coordination state or `memoryd::peer_state` if it must stay daemon-specific. Expose narrow methods equivalent to the current traits and keep `HandlerState` as composition only.
