# Stream I Final Performance Review

Status: Changes requested

## Findings

- Severity 2, crates/memoryd/src/recall/delta.rs:114, Tier 3 and stale-row short-circuiting happens after the hot path has already queried all recall-index rows and built peer candidates. `build_delta_coordination` fetches rows before constructing `SessionContext`/checking `SessionContext::is_tier3`, then evaluates `peer_write_candidates(...).await` as an argument to `RelevanceGate::evaluate` at crates/memoryd/src/recall/delta.rs:120; candidate construction performs one `substrate.read_memory` per row in crates/memoryd/src/recall/source_identity.rs:34. Startup has the same pattern in crates/memoryd/src/recall/startup.rs:193 and crates/memoryd/src/recall/startup.rs:254. Why it matters: the spec requires Tier 3 to return before peer writes are evaluated and keep the Tier 3 path free of per-write cost, while the performance rationale assumes candidate count is bounded by the recency window. In the current wiring, a Tier 3/default-level request and any namespace with many stale passive-recall rows still pays an O(total namespace rows) SQLite query plus per-row memory read before the gate returns empty or filters by `indexed_at`. The p95 bench does not cover this overhead because it starts with an in-memory candidate slice. Suggested fix: resolve the session tier before `delta_peer_candidate_rows`/`startup_peer_candidate_rows` and return no coordination immediately for Tier 3; push the `indexed_at >= now - recency_window` bound into the recall-index query or at least filter rows by `indexed_at` before `peer_source_identity`/`read_memory`; add a hot-path benchmark or integration test with many stale rows that exercises `build_delta_response_with_coordination` and proves Tier 3 does not perform candidate reads.
- Severity 2, crates/memorum-coordination/src/presence.rs:217, The background cleanup task only calls `cleanup_stale_sessions`; it never sweeps expired claim locks, even though `ClaimLockRegistry::sweep_expired_at` exists at crates/memorum-coordination/src/claim_lock.rs:274 and is otherwise only exercised by unit tests. Why it matters: the spec says TTL expiry is released by the claim-lock sweeper running every 60 seconds. Without that runtime sweep, expired Level 2 locks can accumulate in the DashMap until a later lookup/reacquire path touches them, increasing `active_locks`/status scan cost and leaving stale entries in RAM. Suggested fix: wire expired-lock sweeping into the 60-second cleanup task (or spawn a dedicated claim-lock sweeper) and add a `memoryd --test stale_session_cleanup` case proving an expired claim lock is removed after a tick even when the holder session is not stale/present.
- Severity 3, crates/memorum-coordination/src/bin/peer_relevance_bench.rs:172, The p95 evidence is valid only for the isolated in-memory scoring fixture, not the recall hot path. The timer wraps `RelevanceGate::evaluate` over a prebuilt `Vec<PeerWriteCandidate>` at crates/memorum-coordination/src/bin/peer_relevance_bench.rs:178, uses a 16-dimensional embedding vector at crates/memorum-coordination/src/bin/peer_relevance_bench.rs:25, and the evidence file explicitly excludes embedding-worker and candidate acquisition costs. Why it matters: the measured p95 of 0.006917 ms per candidate passes the 5 ms gate, but it does not prove the Final Gate E questions about recall hot-path overhead, source-identity reads, SQLite/index acquisition, or a realistic high-dimensional cosine path. Suggested fix: keep this microbench, but add/record a second release-gate bench for `build_delta_response_with_coordination` with fixed data, many stale rows, 100 in-window candidates, and a realistic embedding dimension or explicit no-embedding production mode; assert both p95 budget and no canonical baseline writes in assert mode.

## Residual risks

- The existing microbench is deterministic and reproducible, but it is not an end-to-end daemon/recall benchmark and was run on unoptimized dev profile; release-mode or CI-profile variance remains unknown.
- DashMap usage in presence and claim-lock registries avoids obvious double-lock/iterator-removal hazards by collecting keys before removal, but large stale maps still create O(n) sweeper/status work until the expired-lock cleanup gap is fixed.
- I did not run the full workspace gate, rustdoc, or long Stream I Task 23 suite; this was the performance lane review only.
