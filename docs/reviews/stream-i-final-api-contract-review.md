# Stream I Final API Contract Review

Status: Changes requested

## Findings

- Severity 2, `crates/memoryd/src/recall/startup.rs:297`, startup peer-update relevance is derived from the candidate peer rows themselves. `startup_session_context` builds `salient_entities` and `salient_paths` from `rows` (`crates/memoryd/src/recall/startup.rs:297-306`), and both same-device and cross-device startup paths pass those same rows to `RelevanceGate::evaluate` (`crates/memoryd/src/recall/startup.rs:245-258`, `crates/memoryd/src/recall/startup.rs:261-281`). That lets arbitrary peer candidate rows self-match through entity/path overlap rather than being judged against the receiving session's startup context. Why it matters: the spec requires startup recall to inject peer activity that is salient to the current session (`docs/specs/stream-i-cross-session-v0.1.md:392-399`) and uses `local_observed_at` only as the recency clock (`docs/specs/stream-i-cross-session-v0.1.md:420-422`), not as a substitute for relevance. This can over-surface same-device and cross-device `<peer-update>` entries in `<memory-recall>`, weakening Stream E startup compatibility and third-party framing. Suggested fix: build the startup `SessionContext` from the receiving session's startup recall/project seeds and selected-id namespace paths, not from the peer candidate rows; add same-device and cross-device startup tests proving unrelated peer writes do not surface.
- Severity 2, `crates/memoryd/src/recall/delta.rs:300`, capped `<peer-presence>` selection is sorted by harness/session id instead of overlap score. After filtering, the code sorts records with `left.harness.cmp(...).then_with(session_id...)` and then takes the first four (`crates/memoryd/src/recall/delta.rs:300-307`). Why it matters: spec §5.2 requires that if more than four live peer sessions overlap, the top four are selected by entity overlap score with the current session and the rest are counted in `capped_peer_presence` (`docs/specs/stream-i-cross-session-v0.1.md:386-390`). Alphabetical ordering can hide the most relevant live collaborators while still reporting a valid-looking `<peer-presence>` block. Suggested fix: compute an entity-overlap score for each active record, sort descending by that score, then use deterministic tie-breakers before applying the cap; add a test with five peers where the alphabetically first peer is not one of the top four by overlap.
- Severity 2, `crates/memorum-coordination/src/presence.rs:262`, `PeerHeartbeatAck.conflicting_claim_locks` is never populated. The ACK DTO exposes `conflicting_claim_locks` (`crates/memorum-coordination/src/protocol.rs:23-29`), but `handle_peer_heartbeat` always returns `conflicting_claim_locks: Vec::new()` (`crates/memorum-coordination/src/presence.rs:262-268`). Why it matters: spec §6.1 defines this field as active claim locks held by other sessions that intersect the heartbeat session's salient entities (`docs/specs/stream-i-cross-session-v0.1.md:473-482`). Level 3 clients therefore receive a stable but misleading empty conflict set even when locks exist. Suggested fix: either populate the field by joining live claim locks to memory/entity projections relevant to the heartbeat, or remove/rename the field and amend the spec/API docs before declaring the protocol stable. Also reconcile the extra `active_peers` ACK field in code/API docs (`crates/memorum-coordination/src/protocol.rs:27`, `docs/api/stream-i-cross-session-api.md:37-43`) with the spec's §6.1 ACK shape, which currently does not list it.
