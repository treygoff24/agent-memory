The mental model

A day in the office, as a loop

Memorum is supposed to work like walking into your office each morning: you pick up where you left off (remembered, not retrieved), things surface as they become relevant, and at day's end you snapshot where things landed for tomorrow. That's a loop. Five of its arrows are cut — and one bug (writes never commit) cuts three of them at once. Each arrow below is labeled with the fix that re-connects it.

The diagnosis

Five cut seams, one keystone

The deepest one is Seam 1: the daemon never git-commits memory writes, which on its own breaks durability, blocks dreaming's clean-tree check, and makes sync meaningless. Every claim below is anchored to real source.

1
Durability — writes never commit to git
The write ladder stops at disk + SQLite + event-log. auto_commit exists but has zero callers in the daemon; the serve loop spawns no commit task. 552 memories sit untracked — the whole git-backing is inert at runtime.

2
Cognition — dreaming can't run, two walls deep
Wall 1: the lease's first step is git fetch origin, which hard-fails with no remote → 6 missed nights. Wall 2: the 552 untracked files from Seam 1 trip the lease's dirty-tree guard. Dreaming is the only automatic capture engine — dead two ways.

3
Capture — there is no ambient capture at all
All three wired hooks are read-only recall. No Stop/SessionEnd hook exists. Capture depends on dreaming (dead) or agents manually calling a write tool (≈never). Six days → zero memories, independent of the dream bug.

4
Recall — the read path is degraded
Recency not relevance; summaries are filename slugs; snippets hardcoded empty; entity-recall starved because the import set entities: [] on everything. Owned by the v3.0 redesign — but it can't improve a store that never grows.

Observability — blind to its own failure
Doctor's health model is clean-tree && ≥1 authed harness. It never reads dream-freshness, sync state, uncommitted-substrate, or recall budget — so it returns green with four seams cut. The merge-conflict notice piled to 96 (append-only queue); the mid-merge flag has zero consumers.

Two decisions (yours)

The forks that steer the build

principle Local-first by default

The most common install — one laptop, no remote — is the case the architecture handled worst: the cognitive engine literally required a network git remote, contradicting the "offline-capable, no cloud required" promise. The commitment: the local daemon is the coordination point; the remote is only the cross-device sync transport, and lights up only when configured. Every network git call gets an origin_remote_configured() guard; absent a remote, it commits locally and no-ops the network as success. Multi-device behavior stays byte-identical when a remote exists.

n=1 Trusted capture

Continuity writes land active/pinned, not in a hand-approved review queue — the friction of approving every session's memory defeats the point. The safety net is the substance/acceptance gate (a hollow snapshot is held back as degraded), not a human in the loop.

n=1 Let the import decay

No batch-enrichment pass over the 552 frozen memories — that polishes the thing the redesign replaces. Let them decay under normal salience while live writes and the continuity engine rebuild from actual use.

The plan

Five fixes that re-connect the loop

Every fix lands in an existing crate — no new stream, no new daemon. F1 is the keystone (it alone clears Seam 2's second wall). Each card notes what the three adversarial reviews changed.

F1 Commit-on-write

A debounced commit worker + a new identity-setting helper that works on an unconfigured-git host, plus a pre-dream flush in the dream process and a repo-level git lock.

Review changed: reuse of auto_commit fails (no git author set); also surfaced a latent bug — sources/web was never staged.

new commit_substrate_writes · server.rs serve loop

F2 Single-device lease

An origin_remote_configured() guard makes fetch_origin/push no-op-and-succeed with no remote. ~6 lines + a flipped test. A configured-but-failing remote still errors (never silently swallowed).

dream/git.rs:75,91 · test dream_lease_election.rs:98

F3 Recovery + dedup

Dedup notifications at the dispatcher on the structured event; a quarantine-resolution CLI re-runs reconcile to clear the wedge; a stranded merge surfaces an operator finding rather than auto-aborting.

Review changed: the queue only stores a flattened string, and nothing re-reconciles after startup — both had to be designed in.

notifications/dispatcher.rs · reconcile.rs

F4 Doctor sees the loop

Doctor takes HandlerState and gains five checks: dream-freshness, sync/quarantine, stale-uncommitted, recall-budget, and capture-freshness (D5) — so the drought can never silently recur. D2/D3 are fatal; the rest advisory.

Review changed: the signature couldn't see recall counters; D3 had to measure stale untracked or it flaps on every write.

handlers/doctor.rs

F5 C0 capture hook

The new SessionEnd closeout hook (the v3.0 headline) lands the continuity write via a distinct governed supersede path that's pinned-active.

Review changed: "governed like a pass-2 candidate" would dump it in the review queue — the promotion carve-out had to be named separately from the governance-check one.

hooks_wire.rs + unwire.rs · v3.0 §5.1

why now v3.0 stands on this

The ambient-recall redesign's continuity engine is "written at closeout, refined at dream-time, committed to git" — it physically can't run until F1, F2, and F5 exist. Redesign-first, done right, means the foundation is Phase 0 of the redesign.
