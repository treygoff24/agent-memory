The verdict in one breath

You diagnosed v0.5 correctly. The cure goes too far.

v0.5 failed by surfacing recency-ranked noise the agent learned to ignore. v1.0 fixes the rendering and ranking — those wins are real and should land. But its risky center, the continuity loop (closeout-writes-startup, the maintained continuity-state object, use-driven decay), inverts the failure mode from noisy irrelevance (ignorable) to confident wrongness — recall the agent trusts, because it's prose, attributed, small, and desk-anchored. That is a worse failure, not a better one.

The cheapest, most trustworthy 80% of this feature needs none of the machinery that creates the hazards. Ship that first.

Every family, working blind, sketched the same continuity-free core: read the desk first (local git), run a relevance-keyed query over the index, render prose recollections on a byte budget. ~40–50% of the build, none of the merge/poisoning/read-only hazards. The fight is only over whether to stop there.

How to read the tags

Four kinds of action

Decide Fix now Verify first Defer

Decide — a genuine product/architecture call only you can make. Fix now — the call is already settled; just correct it, regardless of which architecture you pick. Verify first — confirm one fact about the real harness/repo before sizing the fix. Defer — only bites if you build the continuity object; irrelevant under the simpler path.

The one decision that drives everything

Defer the continuity engine, or build it now?

The synthesis first called this a "convergence." The judge corrected that — it's a real split, and a manual pinned note is not the same artifact as an automated, system-maintained continuity state. This is your call; it decides whether five of the eight blockers are "fix before Phase 3" or simply "not yet relevant."

My recommendation

Option A — Defer & simplify

Codex · Cursor

Ship the continuity-free channel first: desk-first orientation, relevance-gated delta, prose rendering. Prove agents actually use it. Defer the ContinuityState struct, the closeout protocol, and dream-maintenance until their hazards are specified away.

  • Delivers most user-visible value at ~half the build.
  • Sidesteps B2, B3, B5, B7, R5 entirely — they don't exist without the loop.
  • Lets adoption data, not a guess, justify the cathedral.
Why I'd lead here: every Option-B hazard is still unresolved. You can always add continuity once it's earned and safe; you can't easily un-poison startup orientation once a hallucinated "current focus" is pinned.

Option B — Continuity first

Grok · DeepSeek

A prettier recency channel is a cosmetic bandage. The point is remembered-not-retrieved — so build a minimal maintained continuity model (skeleton + volatile) early and treat the hazards as work to do, not reasons to wait.

  • Goes straight at the actual ambition of the redesign.
  • Risk: makes B1, B2, B3, B7 hard pre-requisites, not deferrals.
  • Risk: ships the "confident wrongness" failure mode before the safety gates exist.
Defensible only if you commit to solving the merge, acceptance-gate, read-only, and governance problems before the loop touches startup.

Common ground both camps accept: the desk read ships early, and the full ContinuityState + closeout + dream-maintenance must not ship before B1/B2/B3 are solved. A single human- or agent-authored pinned focus note (via the existing memory_note surface) is the cheapest bridge — minimal continuity for the B camp, no new machinery for the A camp.

Must-fix · the 8 blockers

Design is wrong or will break as specified

B1

Recall claims to be read-only but writes anyway

Fix now
Problem
Recall is declared read-only (inv §12.1), yet PassiveSurfaced/RecallUsed event appends and StagedNote.consumed clearing are writes caused by recall. This was the council's correlated blind spot — no proposer caught it; it surfaced only at critique.
Tradeoff
The synthesis floated "or narrow the invariant to allow deterministic telemetry writes." The judge overrode that: relaxing read-only breaks concurrency safety, testability, and prompt-cache stability. Not worth it.
Reco
Keep recall strictly read-only. Execute all telemetry/consume writes out-of-band — harness-driven, or a separate decoupled async post-render daemon endpoint — with zero side effects on the read path. This is the single highest-confidence finding in the run.
B2

Closeout can poison startup with a hollow snapshot

Defer (A)
Problem
T0 renders continuity-state as authoritative prose. When the agent writes no summary (the common case), fail-open closeout writes a fresh-timestamped but hollow auto-snapshot — so a freshness/updated_at gate is defeated by design.
Tradeoff
Under Option A this can't happen — there's no closeout. Under Option B it's a hard pre-requisite before continuity touches startup.
Reco
If you build continuity: gate on substance, not a timestamp — require non-empty volatile / minimum evidence or an explicit degraded:true flag, add a desk-contradiction check, and frame the block as "remembered, not verified." Ship before Phase 3.
B3

Two devices closing out at once is a merge graveyard

Defer (A)
Problem
Cross-device merge of a single monolithic continuity-state object is unspecified (the spec's own §14.5 open question). Concurrent supersedes from two clones need item-level deterministic merge or the open-loop list silently corrupts.
Tradeoff
Doesn't exist under Option A. Under Option B, "confirm it fits canonical-content equality" is not enough.
Reco
If you build continuity: give open_loops/staged_notes stable item-level IDs, consume events, and version reconciliation; ship a two-device concurrent-closeout merge test as a Phase 3 gate.
B4

The desk read makes blocking subprocess/network calls on the hot path

Fix now
Problem
gh pr view + CI (§8.1) are network calls on the synchronous T0/T2 path — a direct violation of inv §12.10. The judge generalized it: even git status --porcelain can block on .git/index.lock (a concurrent IDE/editor) or take seconds in a monorepo, blowing the 150 ms budget and starving big-repo users of desk context entirely.
Tradeoff
A cached projection adds a background watcher thread; that's cheap next to blocking every session start.
Reco
Make the desk read a daemon-cached background projection (file-watcher → in-memory desk state) read in O(1) by the hooks. No synchronous subprocesses in the hook; drop PR/CI from the sync path or move them to that projection.
B5

"Use" decay cools exactly the recollections that worked

Fix now
Problem
§5.3 decays recollections when the agent doesn't call memory_get on them. But ambient recall succeeds precisely when the agent acts on the self-contained prose without fetching the source. Absence ≠ unused. Phase 4 acceptance makes this live, not deferred.
Tradeoff
None worth keeping — the negative inference is backwards. (Whether memory_get adoption ever yields usable signal is a separate empirical question for telemetry.)
Reco
Make memory_get a positive-only signal. Never infer disuse from its absence. Gate any negative-decay signal on validation data first.
B6

The friction pre-gate is lexical, silent, and untestable

Decide
Problem
A purely lexical gate (errors / keywords / novel entities) silently misses exactly the turns where memory matters most — procedural reuse ("same fix in billing"), social ("reply to Adam"), status continuations. Because the gate is unlogged, the false-negatives are unfalsifiable in production, making T1 untestable. (Judge elevated this from risk to blocker.)
Tradeoff
A permissive gate surfaces more (some noise) but stays measurable; a tight gate stays quiet but blind. You can't tune what you can't see.
Reco
Restrict the pre-gate to obvious no-ops and run the relevance gate on substantive prompts regardless of friction words. Add a miss-signal (a memory_search shortly after a silent T1 = gate miss) and a deterministic self-test (force surfacing on ~1% of turns, keyed by session-id so byte-stability holds) to measure precision without a labeled set.
B7

The governance carve-out is an un-governed poisoning backdoor

Defer (A)
Problem
The §11 carve-out lets system-authored continuity updates skip Stream C dream_source confidence gating — a direct violation of invariant #7 (Stream C stays authoritative). A dream-synthesized hallucination can be promoted straight to pinned, authoritative startup orientation. (Judge elevated this from risk to blocker.)
Tradeoff
Irrelevant under Option A (no system-authored continuity writes). Under Option B it's the actual poisoning vector and must be closed first.
Reco
If you build continuity: do not exempt system writes from Stream C gating. Add claim-level provenance so system-derived vs user-authored is distinguishable in the surface.
B8

Per-turn recall injection can shatter the prompt cache

Verify first
Problem
If a cue-driven T1 block (0–3 memories, changing every turn) lands in the cached prefix, it invalidates prefix caching for the whole session — every turn becomes a full uncached re-eval (latency + compounding cost). This defeats the spec's own cacheability rationale even though each block is byte-deterministic. (Judge — missed by the entire council.)
Verify
Severity depends on where each harness places hook output. If UserPromptSubmit hooks already append at the bottom of context, the prefix is safe. The real defect: the spec doesn't specify injection position at all. Confirm against actual Claude/Codex hook placement before sizing.
Reco
Mandate that dynamic T1/T2 blocks render at the bottom of context (immediately before the latest user prompt), isolated from the static prefix.
Watch-outs · the 8 risks

Real hazards that need an explicit mitigation

R2Cold-start is undefined. The spec never says what T0 does with no continuity-state (first session, or a deleted object). If it falls back to a recency dump, the core bet fails on first contact.Define cold-start explicitly: desk + pinned invariants, no continuity object.Fix now
R4The relevance floor is uncalibrated and gates the whole feature. A wrong threshold either floods or starves.Ship T1 gating behind a flag with a permissive floor; tighten via a labeled Stream H "should-surface?" set before default-on.Fix now
R5Continuity claims never decay. The disuse loop covers recollections, not T0 orientation claims — a wrong "current focus" lives forever.Add staleness/contradiction invalidation for continuity claims (desk mismatch, superseded refs).Defer (A)
R6The strength score is inclusion-polluted. The base ranking's strength term counts RecallHit (inclusion), and the passive path emits none — so Phase 2 eval calibrates on a miscalibrated base.Calibrate explicitly on the structural-only base, or sequence the feedback-signal fix before gating calibration.Fix now
R7Privacy partial-drop can invert meaning. Dropping a fragment can flip a claim ("don't mention X until legal" → "don't mention until legal").A partially-classified object carries a degraded flag and must not silently supersede last-known-good.Fix now
R8Phase 1 is oversold. "This phase alone makes the channel trustworthy" exceeds the evidence — relevance, gating, feedback, and continuity safety are all deferred. Code inspection confirms Phase 1 can't distill "conclusions" for imported-doc chunks.Scope the claim to "stops the noise."Fix now
R9Rephrasing isn't injection safety. Neutralizing imperative prose is incomplete — a poisoned memory still steers as a reported fact ("the standing practice has been to run script X"). (Restored by the judge.)Frame system-derived / low-confidence recall as external evidence ("A prior memory reports…"), not internal fact.Fix now
R10The T2 "desk delta" has nowhere to store its baseline. Disk = second persistence layer; memory = read-only violation; in-process = stateful daemon that drifts on restart. Every option breaks an invariant. (Judge — missed by the council.)Render the current absolute desk state, not a delta. The baseline disappears and both invariants hold.Fix now
Polish · the 7 nits

Small, but worth folding into v1.1

N1§7.1 wording. "Conclusions, not pointers" → "conclusions in body; pointer in attribute for traceability." Don't over-advertise memory_get <ref> into card-catalog behavior. (Keep the opaque ref — do not add a human-readable alias; it diverges across devices and breaks eval-fixture stability.)Fix now
N2Typo: ClosoutSummary → CloseoutSummary (§11).Fix now
N3Dedup is too aggressive. Add a turn-distance threshold so a recollection can re-surface after a meaningful gap / topic shift, instead of permanent intra-session suppression.Fix now
N4Underspecified score ranges. §6.2 needs testable ranges for relevance_to_cue and lesson_boost, or ranking behavior is untestable.Fix now
N5Empty-wrapper exactness. Specify the exact empty-wrapper form per trigger so harness parsers don't invent variants.Fix now
N6Protective-recall coupling. lesson_boost fires only when the friction pre-gate fires — so a missed decision point kills protective recall even when the relevance gate would have selected the lesson. Give it an independent path or document the coupling.Fix now
N7O1 conflates two relevances. "Relevance over recency" bundles cue-relevance (T1, semantic match) and project-salience (T0, stakes ordering) under one slogan; they have different signals and failure modes. Separate them in the objective text.Fix now
The path I'd take

If it were my call

Pick Option A, then sequence the work so nothing unsafe ships before its gate exists.

Correct the cross-cutting bugs first — they're independent of the architecture choice

B1 (read-only writes go out-of-band), B4 (desk read becomes a daemon-cached projection), B5 (memory_get positive-only), B8 (inject at bottom of context, after you verify harness placement), plus R7/R9/R10 and the nits. None of these depend on continuity.

Ship the continuity-free channel — the design all four families sketched

Desk-first T0 orientation (local git, cached), the existing delta path plus a relevance-to-cue term, a conservative pre-gate, prose units on a byte budget, and the discoverable guidance string. Optionally one pinned focus note via memory_note. Resolve R2 (cold-start) and R4 (relevance floor behind a flag) here.

Gate: Stream H shows agents act on surfaced prose — adoption, not just emission.

Make the friction gate measurable before you trust it

B6 — narrow the pre-gate to no-ops, add the miss-signal and deterministic self-test. Calibrate against the labeled set (R4) and the structural-only strength base (R6).

Gate: a precision number you can actually defend, not a vibe.

Only now earn the continuity engine — if adoption data says it's worth it

Solve B2 (substance gate + "remembered, not verified"), B3 (item-level merge + two-device test), B7 (no Stream C carve-out, claim-level provenance), and R5 (continuity-claim invalidation) before the loop touches startup. If adoption never materializes, you saved the whole cathedral.

Gate: B2 + B3 + B7 + R5 all closed before ContinuityState renders at T0.
Net: fix six things now, ship the simple channel, prove it's used, and let that data — not a guess — decide whether the continuity engine is worth its hazards.
