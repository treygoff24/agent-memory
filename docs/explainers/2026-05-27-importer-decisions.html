Q1 / 11

Granularity — what becomes one Memorum memory?

Claude topic files vary wildly (some are 1-fact feedback_*.md at ~20 lines, others are 5KB multi-section dossiers with 6+ distinct facts). Codex Task Groups have rigid structure where facts live in cross-cutting User preferences / Reusable knowledge / Failures bullet sections, all tagged with [Task N]. Going too fine creates parser-fragile decomposition; too coarse loses fact-level revisability.

Q2 / 11

Wiki-link translation

Claude memories contain inline [[wiki_link_target]] cross-references between memories (e.g., Linked: [[project_campaign_intake_spec_wip]], [[feedback_heterogeneous_review_fanout]]). Memorum's frontmatter already has a related: [memory_id] field for this exact purpose.

Q3 / 11

Write primitive per source type

Memorum has three write primitives with different governance behavior: memory_write (full governance + contradiction detection + classification), memory_note (Privacy Filter only, no governance), memory_observe (Stream F dream-substrate fragment, never directly canonical). Q4 / 11

Codex rollout_summaries/ — provenance for Task Group memories

Each Codex Task Group references 1–N rollout summary files in its ### rollout_summary_files section. They're the session-trace artifacts that produced the memory — the closest analog to source-grounding for Codex-origin memories.

Q5 / 11

Non-git cwds — how do we handle Dropbox / scratch dirs?

Memorum's project identity is git remote get-url origin → normalize → SHA256 → proj_<hex>. Codex Task Groups carry applies_to: cwd=/Users/treygoff/Library/CloudStorage/Dropbox/Prospera — Dropbox dirs, no git remote. Memorum has no path-hash fallback; non-git dirs have no project namespace.

Q6 / 11

Entity extraction strategy

Memorum has no body-derived entity extraction — the importer must supply entities explicitly per memory. Codex's ### keywords and Claude's name frontmatter are the obvious source hints; we could also run deterministic regex (file paths, repo names, commit SHAs, PR numbers) or punt and let dreaming consolidate later.

Q7 / 11

Idempotency state file location

Memorum has no content-hash dedup or "skip if exists" mode. Re-running the importer needs its own state file mapping source_path → {memory_id, content_hash, last_imported_at} so we know what's already done.

Q8 / 11

Conflict UX — governance returns refused: contradiction

Governance fires synchronously per write. If memory #51 contradicts already-imported #12, the write returns {status: "refused", reason: "contradiction", existing_id: "..."}. What does the importer do?

Q9 / 11

Bulk throughput strategy

Each memory_write is one socket round-trip + privacy classification + active-memory snapshot reload + governance evaluation. At 1000+ writes that's seconds-per-write. Three real options.

Q10 / 11

Re-import behavior when source has new content

After first import, the user keeps using Claude / Codex. New memories accumulate. Old ones may be edited. User runs memoryd import again. What happens?

Q11 / 11

First-run UX — when does the importer run?

User installs Memorum and runs memoryd serve --init. Should the importer fire automatically, only when explicitly invoked, or be offered as an opt-in during init?
