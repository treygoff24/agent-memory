## Goal

Ship an **opt-in** API-served embedding lane (`gemini-embedding-2`, subject to the D7 bake-off gate) so users who accept the privacy trade can run a **~10–50 MB daemon** instead of the ~1.27 GB local-model footprint. Local Qwen3 stays the default. The API lane must be consent-gated, classification-aware (sensitive content never leaves the machine), fail-open to FTS-only, and quality-gated by the same golden-corpus bench that selected Qwen3.

## Why this is cheap (architecture audit, 2026-07-09; corrections from review pass in ⚠ notes)

- `EmbeddingProvider` (`crates/memoryd/src/embedding/mod.rs:108`) is a 4-method sync trait: `triple()`, `embed_query`, `embed_document`, `embed_documents` (batch default provided). All call sites already run it under `spawn_blocking`, so a blocking HTTP client is the correct shape — no async refactor.
  - ⚠ **Correction (review):** the workspace `reqwest` is async-only (`Cargo.toml:40`, `default-features = false`, no `blocking`). T1.1 enables the `blocking` feature (or adds `ureq`) — a `Cargo.toml` edit with orchestrator-merged lockfile, not "already in-tree."
- Invariant 3 (triple = identity) gives the API lane its own per-triple vec tables for free; no migration hazard against the local index, typed `UnknownEmbeddingTriple`/`DimensionMismatch` errors on misconfiguration.
- The `server.rs:167` gate — `is_fastembed_candle_triple` or bust — is a provider-string dispatch in `spawn_embedding_worker`.
  - ⚠ **Correction (review):** it is NOT the only hard gate. `handlers/doctor.rs:283` hard-codes the same `is_fastembed_candle_triple` check and emits `embedding_provider_unsupported` — must be retired in the same task (T1.2) or a working API lane reports a false-positive doctor finding.
- Lifecycle slot (`EmbeddingProviderSlot`) already handles dormant→loading→active→failed with retry backoff and doctor surfacing; an API provider's "load" is constructing an HTTP client + a credential check, so the same state machine works with a trivial loader.
- Fail-open exists end-to-end: query-time embed timeout → `embedding_timeout` degrade marker → FTS-only; `MEMORUM_DISABLE_EMBEDDING_WORKER` path; doctor findings for load failure.
- ⚠ **Correction (review):** `EmbeddingError` has only `Load`/`Inference`/`DimensionMismatch` (`mod.rs:83`). The auth/rate-limit/transport/contract taxonomy T1.1 needs means new variants in `mod.rs` — an additive edit to a file shared with the local provider, owned by T1.1.

## Decisions

- **D1 — Vendor-specific provider, not a generic "OpenAI-compatible" shim.** First implementation targets one vendor's API exactly. Note (codex cross-check): gemini-embedding-2 uses **prompt prefixes**, not `task_type` params — the asymmetric query/document mapping is provider-impl logic, exactly why a generic shim would silently degrade retrieval. Second vendor, if ever, is a second small provider.
- **D2 — API key lives in device-local runtime state, never `config.yaml`.** Same rationale as invariant 4 (device IDs): `config.yaml` syncs across clones. Resolution order: env var (`MEMORUM_GEMINI_API_KEY`) → runtime-state key file (0600). Keychain integration deferred.
- **D3 — Privacy fence is classification-driven, not just a consent checkbox.** Even with the lane enabled, content whose persisted sensitivity tier is not on the API-eligible allowlist is **never sent to the API** — those memories remain FTS-only under the API lane (no local-model fallback in v1 — dual-provider residency would reintroduce the footprint the user opted out of). Consent covers: memory/chunk text of API-eligible memories, and **query text** (recall queries transit the API too — the consent language must say so explicitly, and R1's test obligations distinguish the two: query transit is consent-covered by design; memory-text transit is fence-governed).
- **D4 — Switching lanes = full re-embed of eligible memories.** Triple change enqueues embedding jobs for all active memories via the existing reconcile machinery — which is itself a fence point (see D9): reconcile must apply the same eligibility predicate as the drain fetch, or the lane switch bulk-leaks. Old triple's tables are left in place (cheap, enables switch-back); a `doctor` note reports orphaned triples with a cleanup hint.
- **D5 — Gates never touch the network.** All tests run against a mock HTTP server (or the FixtureProvider); real-API runs are off-gate manual, same policy as real-model bench runs.
- **D6 — Query-time embed timeout gets a lane-specific default.** Local default 50 ms cannot survive an HTTP round trip; API lane defaults to ~250 ms. `embed_timeout_ms` is currently a single global field (`recall/config.rs:72`) — T2.3 specifies where lane-aware selection lives (resolved at provider activation: the effective default keys off the active triple's provider string; explicit config always wins). Still fail-open to FTS on timeout. p95 budgets renegotiated with measured numbers per the Stream E amendment discipline, never silently busted.
- **D7 — Model choice is empirical.** The golden-corpus bench (`fixtures/golden/_embed_bench/`) produces the decision. Ship gate: candidate must beat or match Qwen3-0.6B on **trap-rate@5** and **abstention gap** — nDCG wins alone do not qualify (that's how EmbeddingGemma lost).
- **D8 — Synchronous list batching only; no vendor file-based Batch APIs.** Several vendors' async batch/file endpoints retain uploaded files or job state under separate (non-ZDR) terms. Our drain already batches synchronously via `embed_documents` — codify it: memory text never transits a vendor file store. (Cost: forgo batch-tier discounts; acceptable at personal scale.)
- **D9 — Every embed ingress is a fence point (NEW, from review pass).** The drain fetch is not a "single choke point." Under an API triple there are **four** paths that hand text to the active provider, and each is individually fenced:
  1. **Drain fetch** (`Index::pending_embedding_jobs`, `index/query.rs:461`) — SQL predicate on the `memories` join (T2.1a).
  2. **Reconcile enqueue** (`reconcile_active_embedding_jobs`, `index/embedding.rs:146`) — currently selects every chunk missing a vector, including encrypted `safe_body` projections (`api/mod.rs:554` sets `index_body = true` on the masked projection). Same predicate as (1) (T2.1a).
  3. **Governance write-time similarity** (`handlers/governance/pipeline.rs:123` → `policy.rs:397` `embed_query(candidate.claim())`) — embeds the **plaintext claim of every governed write, before encryption, with no sensitivity gate**. Under an API triple, sensitive candidates skip similarity via the existing `SimilarityResolution::degraded` path (T2.0).
  4. **Query-time recall embed** (`recall/hybrid.rs:81`) — user/agent message text. **Consent-covered by design (D3), not fence-blocked**; named explicitly in consent copy (Q5).
  The fence predicate operates on the **persisted `memories.sensitivity` tier** (`index/schema.rs:26`), NOT on `ClassificationOutcome` — `RequiresEncryption` is a runtime value that is never persisted (invariant 1). T2.1a defines the concrete API-eligible tier allowlist from the `memory-privacy` tier enum; the R1 test matrix enumerates **every persisted sensitivity tier**, plus the encrypted-projection case, against a mock server that fails the test on any unexpected request body.
- **D10 — Fenced-out jobs get a terminal state, not an SQL blindspot (NEW, from review pass).** A fetch-side `WHERE` alone wedges the drain: the worker loops on `pending_embedding_job_count()` (`worker.rs:102`), which is an unfiltered `COUNT(*)` (`index/vector.rs:89`) — held-local rows would keep the count > 0 forever (perpetual `ensure_loaded`, doctor backlog noise, no honest accounting). Fenced-out jobs are resolved to an explicit held-local terminal state (or excluded from the active count and reported by a dedicated count), so `status`/`doctor` can truthfully say "N memories held local-only under API lane."

## Risks

- **R1 — Privacy fence gaps.** A single sensitive chunk leaking to the API is a trust-model breach, not a bug. The v0.1 "single choke point" framing was **wrong** — the review pass found two unfenced paths (governance write-time similarity; reconcile enqueue of encrypted projections). Mitigation now: D9 enumerates all four embed ingress points and fences three (the fourth, query text, is consent-covered and named in Q5 copy); the test matrix covers every **persisted sensitivity tier** (not `ClassificationOutcome` variants — those aren't persisted) plus the encrypted-projection case; mock server fails on any unexpected request; cross-family adversarial review is a mandatory Wave-2 gate.
- **R2 — Vendor drift.** API models get deprecated/re-versioned; a silent server-side model swap changes vector space. Mitigation: triple pins the exact model string; dimension check catches gross drift; doctor can't catch subtle drift — document as a known limitation of any API lane.
- **R3 — Quality regression on our failure-mode metrics** despite better MTEB numbers. Mitigation: D7 ship gate is non-negotiable.
- **R4 — Cost surprise on big imports.** A 10k-memory import is ~ millions of tokens. Mitigation: the T3.1 cost estimate (explicit deliverable, not aspiration) shown in the consent prompt before enqueueing re-embed.
- **R5 — Latency p95 regressions on recall.** Mitigation: D6 timeout + degrade marker; measured renegotiation only.
