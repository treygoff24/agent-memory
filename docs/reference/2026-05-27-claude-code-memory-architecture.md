# Claude Code memory system — architecture report (2026-05-27)

This report describes the default Claude Code memory behavior a normal user encounters without custom profile trees, custom `CLAUDE_CONFIG_DIR`, or custom `autoMemoryDirectory`. Claude Code's public docs now distinguish two memory surfaces: user-authored `CLAUDE.md` instruction files and Claude-authored auto memory. The most important default-storage fact is that auto memory is not kept next to `CLAUDE.md`; it lives under Claude Code's global configuration tree at `~/.claude/projects/<project>/memory/`, where `<project>` is derived from the repository or project root ([Claude Code memory docs](https://code.claude.com/docs/en/memory#storage-location); [Claude Code directory reference](https://code.claude.com/docs/en/claude-directory)).

The findings below intentionally separate official documentation from public reports and unresolved behavior. Anthropic's docs and changelog are authoritative for paths, command names, and stated lifecycle behavior; GitHub issues and third-party walkthroughs are useful evidence of user-observed behavior but are not treated as a specification.

## Storage layout

Claude Code stores default auto memory per project under the user's Claude Code configuration directory. The official memory page says each project gets `~/.claude/projects/<project>/memory/`; the `<project>` component is derived from the git repository, so subdirectories and git worktrees belonging to the same repository share one auto-memory directory, while a directory outside git uses the project root instead ([memory docs](https://code.claude.com/docs/en/memory#storage-location)). The directory reference repeats that `projects/<project>/memory/` is a global-only location for auto memory, separate from project-local `.claude/` files ([directory reference](https://code.claude.com/docs/en/claude-directory#file-reference)). On Windows, the same docs define `~/.claude` as `%USERPROFILE%\.claude`; if `CLAUDE_CONFIG_DIR` is set, every `~/.claude` path on the directory page is relocated under that directory ([directory reference](https://code.claude.com/docs/en/claude-directory#path-conventions); [environment variables](https://code.claude.com/docs/en/env-vars)). In the default install, without `CLAUDE_CONFIG_DIR`, the practical paths are therefore:

- macOS/Linux: `~/.claude/projects/<project>/memory/` for auto memory ([memory docs](https://code.claude.com/docs/en/memory#storage-location)).
- Windows: `%USERPROFILE%\.claude\projects\<project>\memory\` for auto memory, because `~/.claude` resolves to `%USERPROFILE%\.claude` on Windows ([directory reference](https://code.claude.com/docs/en/claude-directory#path-conventions)).

The default `CLAUDE.md` storage layout is different. Claude Code supports organization-managed, user, project, and local `CLAUDE.md` files. The official memory page lists managed-policy locations as `/Library/Application Support/ClaudeCode/CLAUDE.md` on macOS, `/etc/claude-code/CLAUDE.md` on Linux and WSL, and `C:\Program Files\ClaudeCode\CLAUDE.md` on Windows; user instructions live at `~/.claude/CLAUDE.md`; project instructions live at `./CLAUDE.md` or `./.claude/CLAUDE.md`; local personal project instructions live at `./CLAUDE.local.md` ([memory docs](https://code.claude.com/docs/en/memory#choose-where-to-put-claude-md-files)). The settings page gives the same scope model for settings and notes that Windows maps `~/.claude` to `%USERPROFILE%\.claude` ([settings docs](https://code.claude.com/docs/en/settings#what-uses-scopes)).

Claude Code also has subagent memory directories, but those are not the main-session auto memory store. The directory reference says subagents with `memory: project` use `.claude/agent-memory/`, `memory: local` uses `.claude/agent-memory-local/`, and `memory: user` uses `~/.claude/agent-memory/`; it explicitly describes this as distinct from main-session auto memory under `~/.claude/projects/` ([directory reference](https://code.claude.com/docs/en/claude-directory)). The subagent docs similarly state that a subagent with persistent memory gets the first 200 lines or 25KB of its own `MEMORY.md`, not the main session's auto memory ([subagent docs](https://code.claude.com/docs/en/sub-agents#enable-persistent-memory)).

## File format and schema

The official auto-memory directory contains a `MEMORY.md` entrypoint and optional topic files. Anthropic's memory page shows the layout as `~/.claude/projects/<project>/memory/MEMORY.md` plus topic files such as `debugging.md`, `api-conventions.md`, and `build-commands.md`; it describes `MEMORY.md` as the concise index loaded into every session and topic files as optional files Claude can create and read when relevant ([memory docs](https://code.claude.com/docs/en/memory#storage-location)). The same page states that auto memory files are plain Markdown and that users can edit or delete them at any time ([memory docs](https://code.claude.com/docs/en/memory#manual-edits)).

The only formally documented startup-read limit is for `MEMORY.md`. Claude Code loads the first 200 lines or the first 25KB of `MEMORY.md`, whichever comes first, at the start of every conversation; content beyond that threshold is not loaded at startup. The docs say this limit applies only to `MEMORY.md`, and that `CLAUDE.md` files are loaded in full even though shorter files are recommended for adherence ([memory docs](https://code.claude.com/docs/en/memory#what-gets-loaded)). The changelog records the 25KB cap as an added behavior: `MEMORY.md` previously had a 200-line limit, and a March 2026 release added the 25KB truncation as well ([Claude Code changelog](https://code.claude.com/docs/en/changelog)).

The directory reference provides the clearest public example of the auto-memory file shapes. Its `MEMORY.md` example is a Markdown index with a top-level heading, topic categories, and relative Markdown links to topic files with short descriptions; its topic-file example includes YAML frontmatter with `name`, `description`, and `type: reference`, followed by Markdown body content ([directory reference](https://code.claude.com/docs/en/claude-directory)). The official memory page itself does not specify a required frontmatter schema or an enum of allowed `type` values for main-session auto-memory topic files. Therefore, `MEMORY.md` as a Markdown index and topic `.md` files as plain Markdown are documented; a mandatory topic-file frontmatter schema is not.

Anthropic's March 13, 2026 changelog entry says Claude Code added last-modified timestamps to memory files so Claude can reason about fresh versus stale memories ([changelog, v2.1.75](https://code.claude.com/docs/en/changelog)). The public docs available today do not specify whether those timestamps are written into Markdown frontmatter, stored as filesystem metadata, or represented in another internal field. That is an open schema question, not a basis for inventing a frontmatter field.

No official docs found describe a SQLite database, FTS index, embeddings index, or separate auxiliary index for default local auto memory. The documented index is the `MEMORY.md` file itself: Claude uses it to track what is stored where, loads its beginning at startup, and reads topic files on demand ([memory docs](https://code.claude.com/docs/en/memory#storage-location); [directory reference](https://code.claude.com/docs/en/claude-directory)). Claude Code does store transcripts as JSONL under `~/.claude/projects/<project>/<session-id>.jsonl`, but the sessions page identifies those as conversation transcripts, not memory files ([sessions docs](https://code.claude.com/docs/en/sessions#export-a-conversation)).

## CLAUDE.md vs. the memory tool

Claude Code's current docs make `CLAUDE.md` and auto memory separate, complementary mechanisms. `CLAUDE.md` files are user-written Markdown instructions: coding standards, workflows, project architecture, and rules the user wants Claude to follow. Auto memory is Claude-written Markdown: build commands, debugging insights, preferences, and patterns Claude discovers while working ([memory docs](https://code.claude.com/docs/en/memory#claude-md-vs-auto-memory)). Both are loaded as context at the start of a conversation, but they differ in authorship, scope, and size behavior. `CLAUDE.md` can be project-, user-, or organization-scoped and is loaded in full; auto memory is per repository, shared across worktrees, and startup loading is capped to the first 200 lines or 25KB of `MEMORY.md` ([memory docs](https://code.claude.com/docs/en/memory#claude-md-vs-auto-memory)).

`CLAUDE.md` lives in visible instruction-file locations: `~/.claude/CLAUDE.md`, `./CLAUDE.md`, `./.claude/CLAUDE.md`, `./CLAUDE.local.md`, and the managed-policy paths listed above ([memory docs](https://code.claude.com/docs/en/memory#choose-where-to-put-claude-md-files)). Auto memory lives under `~/.claude/projects/<project>/memory/` and is machine-local; it is not committed with the project by default and is not shared across machines or cloud environments ([memory docs](https://code.claude.com/docs/en/memory#storage-location)). Project rules under `.claude/rules/*.md` and user rules under `~/.claude/rules/` are another instruction surface, not auto memory; the directory reference lists `rules/*.md` as topic-scoped instructions and `projects/<project>/memory/` as auto memory ([directory reference](https://code.claude.com/docs/en/claude-directory#file-reference)).

Anthropic also documents an API-level `memory` tool for applications using the Messages API. That tool exposes a client-side `/memories` directory through commands such as `view`, `create`, `str_replace`, `insert`, `delete`, and `rename`; the docs explicitly say developers control storage and retrieval by implementing the tool ([API memory tool docs](https://platform.claude.com/docs/en/agents-and-tools/tool-use/memory-tool)). Anthropic separately documents Managed Agents memory stores mounted at `/mnt/memory/`, backed by API objects with versioning and hashes ([Managed Agents memory docs](https://platform.claude.com/docs/en/managed-agents/memory); [Memory Stores API](https://platform.claude.com/docs/en/api/beta/memory_stores/memories)). Those API systems are distinct from the default Claude Code CLI auto-memory directory unless an application or integration explicitly connects them; Claude Code's CLI memory docs describe local Markdown files, not the API memory-store object model ([memory docs](https://code.claude.com/docs/en/memory#storage-location)).

## Write triggers

Auto memory is on by default in current Claude Code releases. The memory page says auto memory requires Claude Code v2.1.59 or later, is enabled by default, can be toggled in `/memory`, can be disabled with `autoMemoryEnabled`, and can be disabled through `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1` ([memory docs](https://code.claude.com/docs/en/memory#enable-or-disable-auto-memory); [environment variables](https://code.claude.com/docs/en/env-vars)). The settings page adds that when `autoMemoryEnabled` is false, Claude does not read from or write to the auto-memory directory, and the environment variable can disable it through settings `env` ([settings docs](https://code.claude.com/docs/en/settings#available-settings)).

The normal default write trigger is Claude's own judgment during a session. The official memory page says Claude saves notes for itself as it works, including build commands, debugging insights, architecture notes, code-style preferences, and workflow habits; it also says Claude does not save something every session and decides what is worth remembering based on future usefulness ([memory docs](https://code.claude.com/docs/en/memory#auto-memory)). The same page says Claude reads and writes memory files during the session and that the interface shows banners such as "Writing memory" and "Recalled memory" when Claude is actively updating or reading from the auto-memory directory ([memory docs](https://code.claude.com/docs/en/memory#runtime-behavior)).

User prompts can explicitly cause memory writes. The memory page says that when the user asks Claude to remember something, such as a tool preference or a local test dependency, Claude saves it to auto memory; if the user wants an instruction in `CLAUDE.md` instead, they should ask Claude to add it to `CLAUDE.md` or edit the file through `/memory` ([memory docs](https://code.claude.com/docs/en/memory#view-and-edit-with-memory)). `/memory` itself is the visible management command: the commands reference says `/memory` edits `CLAUDE.md` memory files, enables or disables auto memory, and views auto-memory entries ([commands reference](https://code.claude.com/docs/en/commands)).

`/init` is a `CLAUDE.md` trigger, not an auto-memory trigger in the public docs. The memory page says `/init` generates a starting `CLAUDE.md`; if a `CLAUDE.md` already exists, it suggests improvements instead of overwriting it. With `CLAUDE_CODE_NEW_INIT=1`, `/init` becomes an interactive multi-phase flow that can set up `CLAUDE.md`, skills, and hooks ([memory docs](https://code.claude.com/docs/en/memory#set-up-a-project-claude-md); [commands reference](https://code.claude.com/docs/en/commands)).

The historical `#` quick-add shortcut should not be treated as a current default write trigger. The changelog says Claude Code v0.2.54 added a shortcut that quickly added to Memory when a message started with `#`, and v0.2.66 fixed accidental memory or bash mode activation when pasting ([changelog](https://code.claude.com/docs/en/changelog)). But the v2.0.70 changelog says the `#` shortcut for quick memory entry was removed and tells users to ask Claude to edit `CLAUDE.md` instead ([changelog, v2.0.70](https://code.claude.com/docs/en/changelog)). Current command docs list `/memory` but do not document `#` as a command ([commands reference](https://code.claude.com/docs/en/commands)).

The docs do not say that `/clear`, `/compact`, hooks, or end-of-session compaction automatically write auto memory in the default install. `/clear` empties the context window while preserving the previous conversation for resume; `/compact` replaces earlier conversation with a summary; `/export` exports the current conversation as plain text ([sessions docs](https://code.claude.com/docs/en/sessions#manage-long-conversations)). The context-window guide says after compaction the project root `CLAUDE.md`, unscoped rules, and auto memory are re-injected from disk; that is a load behavior, not a documented memory write trigger ([context-window guide](https://code.claude.com/docs/en/context-window#after-compaction)). Hooks can run user-defined commands at events such as tool use or session events, but the hooks docs describe a user-configurable extension mechanism, not a default auto-memory writer ([hooks docs](https://code.claude.com/docs/en/hooks)).

## Hierarchy and tier composition

For `CLAUDE.md`, Claude Code uses a hierarchy from broadest scope to most specific scope. The memory docs list managed policy, user instructions, project instructions, and local instructions in load order, with project instructions appearing in context after user instructions ([memory docs](https://code.claude.com/docs/en/memory#choose-where-to-put-claude-md-files)). It also says `CLAUDE.md` and `CLAUDE.local.md` files in the directory hierarchy above the working directory are loaded in full at launch, while files in subdirectories load on demand when Claude reads files in those directories ([memory docs](https://code.claude.com/docs/en/memory#choose-where-to-put-claude-md-files)). Imported files use `@path/to/import` syntax, resolve relative to the importing file, and can recurse up to four hops ([memory docs](https://code.claude.com/docs/en/memory#import-additional-files)).

`CLAUDE.md` composition is additive context, not a strict override system. Anthropic's memory page describes these files as context rather than enforced configuration, and it warns that contradictory rules can cause inconsistent behavior ([memory docs](https://code.claude.com/docs/en/memory#claude-md-vs-auto-memory); [memory docs](https://code.claude.com/docs/en/memory#write-effective-instructions)). Settings do have precedence rules: managed settings override command-line, local, project, and user settings; command-line overrides local, project, and user; local overrides project and user; project overrides user ([settings docs](https://code.claude.com/docs/en/settings#how-scopes-interact)). That precedence applies to settings, not to `CLAUDE.md` content.

Auto memory has a narrower default hierarchy. The main-session auto-memory directory is per repository and shared across worktrees and subdirectories in the same repository; outside git, the project root is used instead ([memory docs](https://code.claude.com/docs/en/memory#storage-location)). It is machine-local and not shared across machines or cloud environments ([memory docs](https://code.claude.com/docs/en/memory#storage-location)). There is no documented global main-session auto-memory file equivalent to `~/.claude/CLAUDE.md`; the global `~/.claude/projects/<project>/memory/` directory is global only in the sense that it lives in the Claude Code config tree, but the memory contents are project-keyed ([directory reference](https://code.claude.com/docs/en/claude-directory#file-reference)).

`autoMemoryDirectory` can redirect storage, but it is not part of the default install. Anthropic says `autoMemoryDirectory` must be absolute or `~/`-prefixed and is accepted from policy settings, user settings, and `--settings`; it is not accepted from project or local settings because a cloned repository could otherwise redirect memory writes to sensitive locations ([memory docs](https://code.claude.com/docs/en/memory#storage-location); [settings docs](https://code.claude.com/docs/en/settings#available-settings)). `autoMemoryEnabled`, by contrast, is a setting that can turn reading and writing of the auto-memory directory off; the memory docs show it in project settings, and the settings reference lists the default as true ([memory docs](https://code.claude.com/docs/en/memory#enable-or-disable-auto-memory); [settings docs](https://code.claude.com/docs/en/settings#available-settings)).

## Lifecycle: eviction, deduplication, summarization

The documented lifecycle for default auto memory is persistence as plain Markdown until the user, Claude, or a project-state purge changes it. The memory docs say auto-memory files are plain Markdown that users can edit or delete at any time ([memory docs](https://code.claude.com/docs/en/memory#manual-edits)). The directory reference says `claude project purge` deletes the state Claude Code holds for a project, including transcripts and auto memory under `projects/` ([directory reference](https://code.claude.com/docs/en/claude-directory#clean-up-project-state)); the changelog says `claude project purge [path]` was added on May 1, 2026 to delete all Claude Code state for a project, including transcripts, tasks, file history, and a config entry ([changelog, v2.1.126](https://code.claude.com/docs/en/changelog)).

The docs do not specify an eviction policy for auto-memory topic files. They specify startup truncation of `MEMORY.md` to the first 200 lines or 25KB and say Claude keeps `MEMORY.md` concise by moving detailed notes into separate topic files ([memory docs](https://code.claude.com/docs/en/memory#what-gets-loaded)). That is summarization or organization of the index, not documented deletion. The context-window guide confirms that auto memory is part of startup context and is re-injected from disk after compaction ([context-window guide](https://code.claude.com/docs/en/context-window#after-compaction)).

The sessions page documents cleanup for transcripts, not auto memory: transcripts are stored as JSONL under `~/.claude/projects/<project>/<session-id>.jsonl` and removed after 30 days by default unless `cleanupPeriodDays` changes ([sessions docs](https://code.claude.com/docs/en/sessions#export-a-conversation)). The directory reference separately says `claude project purge` removes auto memory under `projects/`, but the public cleanup text does not clearly say that automatic transcript cleanup evicts auto-memory files ([directory reference](https://code.claude.com/docs/en/claude-directory#clean-up-project-state)).

No official default-CLI docs found describe deduplication, semantic merge, version history, or conflict history for local auto memory. Managed Agents memory stores are different: Anthropic's Managed Agents API docs describe memory objects with immutable versions, `content_hash`, and 30-day version retention ([Managed Agents memory docs](https://platform.claude.com/docs/en/managed-agents/memory); [Memory Stores API](https://platform.claude.com/docs/en/api/beta/memory_stores/memories)). That API object lifecycle should not be projected onto Claude Code's local Markdown auto-memory files, because the CLI memory docs describe filesystem Markdown files and do not mention managed memory-store versions ([memory docs](https://code.claude.com/docs/en/memory#storage-location)).

## User-visible commands

The central command is `/memory`. The current commands reference says `/memory` edits `CLAUDE.md` memory files, enables or disables auto memory, and views auto-memory entries ([commands reference](https://code.claude.com/docs/en/commands)). The memory page gives more detail: `/memory` lists all `CLAUDE.md`, `CLAUDE.local.md`, and rules files loaded in the current session, lets the user toggle auto memory, provides a link to open the auto-memory folder, and opens selected files in the editor ([memory docs](https://code.claude.com/docs/en/memory#view-and-edit-with-memory)).

Other user-visible commands touch adjacent memory surfaces. `/init` creates or improves a project `CLAUDE.md`, and the commands page recommends running `/init` in the first session in a repository and then `/memory` to refine it ([commands reference](https://code.claude.com/docs/en/commands); [memory docs](https://code.claude.com/docs/en/memory#set-up-a-project-claude-md)). `/context` shows context usage and optimization suggestions, including memory bloat warnings after the March 12, 2026 release that added context suggestions ([changelog, v2.1.74](https://code.claude.com/docs/en/changelog); [context-window guide](https://code.claude.com/docs/en/context-window)). `/clear` and `/compact` manage conversation context, not memory files; `/export` exports the current conversation as plain text, not the memory directory ([sessions docs](https://code.claude.com/docs/en/sessions#manage-long-conversations); [commands reference](https://code.claude.com/docs/en/commands)).

At the filesystem surface, users are expected to read, edit, or delete Markdown files. The auto-memory docs explicitly say the files are plain Markdown and can be edited or deleted any time, and the `CLAUDE.md` docs describe project, user, local, and imported instruction files as normal Markdown files ([memory docs](https://code.claude.com/docs/en/memory#manual-edits); [memory docs](https://code.claude.com/docs/en/memory#import-additional-files)). `claude project purge` is the visible cleanup command for deleting Claude Code state for a project, including auto memory under `projects/` ([directory reference](https://code.claude.com/docs/en/claude-directory#clean-up-project-state)). `/release-notes` is user-visible for viewing the changelog in the CLI, and the docs also publish the changelog at code.claude.com and GitHub ([commands reference](https://code.claude.com/docs/en/commands); [GitHub changelog](https://github.com/anthropics/claude-code/blob/main/CHANGELOG.md)).

## Export / backup / portability

No official source found documents a Claude Code memory-specific `--export`, `/export-memory`, or backup command for local auto memory. The documented `/export [filename]` command exports the current conversation as plain text, either to a file or to a copy/save dialog, and the sessions page describes the same command as exporting messages and tool outputs from the current conversation ([commands reference](https://code.claude.com/docs/en/commands); [sessions docs](https://code.claude.com/docs/en/sessions#export-a-conversation)). That is a transcript export, not a memory export.

The official portable representation of default local auto memory is the on-disk Markdown directory itself. Anthropic says auto memory files are plain Markdown, machine-local, not shared across machines or cloud environments, and stored under `~/.claude/projects/<project>/memory/` unless `autoMemoryDirectory` is configured ([memory docs](https://code.claude.com/docs/en/memory#storage-location); [memory docs](https://code.claude.com/docs/en/memory#manual-edits)). For `CLAUDE.md`, portability is the file itself: project `CLAUDE.md` and `.claude/CLAUDE.md` are normally shared through source control, local `CLAUDE.local.md` should be gitignored, and user instructions live in `~/.claude/CLAUDE.md` ([memory docs](https://code.claude.com/docs/en/memory#choose-where-to-put-claude-md-files)).

Managed Agents memory has an API surface for listing and retrieving memory objects and their versions, but that is not the default Claude Code CLI local memory store. The Managed Agents docs describe memory stores as workspace-scoped documents mounted at `/mnt/memory/` and the Memory Stores API documents object retrieval, content hashes, and versions ([Managed Agents memory docs](https://platform.claude.com/docs/en/managed-agents/memory); [Memory Stores API](https://platform.claude.com/docs/en/api/beta/memory_stores/memories)). The Claude Code CLI docs for default auto memory do not advertise that API as a backup mechanism for local `~/.claude/projects/<project>/memory/` Markdown files ([memory docs](https://code.claude.com/docs/en/memory#storage-location)).

## Concurrent-write semantics

Official docs do not define concurrent-write semantics for auto memory. The memory page says all worktrees and subdirectories within the same git repository share one auto-memory directory, and the February 28, 2026 changelog confirms project configs and auto memory were changed to be shared across git worktrees of the same repository ([memory docs](https://code.claude.com/docs/en/memory#storage-location); [changelog, v2.1.63](https://code.claude.com/docs/en/changelog)). That establishes shared storage. It does not specify file locking, atomic writes, merge behavior, conflict files, or last-writer-wins behavior.

The broader Claude Code docs do mention concurrency in adjacent areas. The sessions page says if the same conversation is resumed in multiple terminals without forking, messages from both terminals interleave into one transcript ([sessions docs](https://code.claude.com/docs/en/sessions#branch-or-fork-a-conversation)). The changelog says v2.1.61 fixed concurrent writes corrupting a config file on Windows and v2.1.126 fixed a rare race where a concurrent credential write could clear a valid OAuth refresh token ([changelog](https://code.claude.com/docs/en/changelog)). Those entries show Anthropic has addressed concurrency defects in other local files, but they do not specify auto-memory locking.

Public issue reports raise concurrency and path-keying concerns, but they are not a contract. For example, a public GitHub issue reports unwanted divergence of memories between repo and worktree paths before the worktree-sharing change ([GitHub issue #44130](https://github.com/anthropics/claude-code/issues/44130)), and an issue mirror describes a suspected concurrent write race on shared `MEMORY.md` ([StepCodex issue mirror](https://www.stepcodex.com/en/issue/bug-concurrent-write-race-on-shared)). These reports are useful as cautionary field evidence, but the official answer is simply that no current public Claude Code doc specifies concurrent-write semantics for default local auto memory.

## Recent history and changelog

The memory system changed substantially across late 2025 and early 2026. As of this report date, the published changelog's latest entry is Claude Code v2.1.152 on May 27, 2026; that release does not announce a storage-model change for auto memory, but it does mention `/usage` scanning large session files with streaming reads so memory usage stays flat ([changelog, v2.1.152](https://code.claude.com/docs/en/changelog); [GitHub release v2.1.152](https://github.com/anthropics/claude-code/releases/tag/v2.1.152)).

The last six months of changelog history show these memory-relevant changes:

- February 5, 2026, v2.1.32: Claude Code began automatically recording and recalling memories as it worked, according to the changelog entry ([changelog](https://code.claude.com/docs/en/changelog)).
- February 6, 2026, v2.1.33: subagent definitions gained a `memory` frontmatter field with `user`, `project`, or `local` scope, enabling persistent subagent memory ([changelog](https://code.claude.com/docs/en/changelog); [subagent docs](https://code.claude.com/docs/en/sub-agents#enable-persistent-memory)).
- February 26, 2026, v2.1.59: Claude Code introduced auto memory as documented today: Claude automatically saves useful context to auto memory and users manage it with `/memory` ([changelog, v2.1.59](https://code.claude.com/docs/en/changelog); [memory docs](https://code.claude.com/docs/en/memory#auto-memory)).
- February 28, 2026, v2.1.63: project configs and auto memory were changed to be shared across git worktrees of the same repository ([changelog, v2.1.63](https://code.claude.com/docs/en/changelog)).
- March 12, 2026, v2.1.74: `autoMemoryDirectory` was added so users or policy could configure a custom auto-memory storage directory; current docs restrict that setting to policy, user settings, and `--settings`, not project or local settings ([changelog, v2.1.74](https://code.claude.com/docs/en/changelog); [settings docs](https://code.claude.com/docs/en/settings#available-settings)).
- March 13, 2026, v2.1.75: last-modified timestamps were added to memory files ([changelog, v2.1.75](https://code.claude.com/docs/en/changelog)).
- March 20, 2026, v2.1.81: the `--bare` flag was introduced for scripted calls and auto memory was fully disabled in that mode; current environment-variable docs also say `CLAUDE_CODE_SIMPLE`/`--bare` disables auto memory and `CLAUDE.md` auto-discovery ([changelog, v2.1.81](https://code.claude.com/docs/en/changelog); [environment variables](https://code.claude.com/docs/en/env-vars)).
- Later March 2026: the `MEMORY.md` startup index cap became both first 200 lines and 25KB, not just a line count ([changelog](https://code.claude.com/docs/en/changelog); [memory docs](https://code.claude.com/docs/en/memory#what-gets-loaded)).
- May 1, 2026, v2.1.126: `claude project purge [path]` was added for deleting Claude Code state for a project, and the directory reference says that state includes auto memory under `projects/` ([changelog, v2.1.126](https://code.claude.com/docs/en/changelog); [directory reference](https://code.claude.com/docs/en/claude-directory#clean-up-project-state)).
- Around spring 2026, the changelog added a UI improvement where memory filenames in the "Saved N memories" notice highlight on hover and open on click ([changelog](https://code.claude.com/docs/en/changelog)).

Older history also matters for interpreting stale user advice. The changelog says `/memory` gained direct editing of all imported memory files in v1.0.94 on August 27, 2025 ([changelog, v1.0.94](https://code.claude.com/docs/en/changelog)). It also says the `#` quick memory shortcut was added in v0.2.54 on April 2, 2025, but removed in v2.0.70 on December 15, 2025 with guidance to ask Claude to edit `CLAUDE.md` instead ([changelog, v0.2.54 and v2.0.70](https://code.claude.com/docs/en/changelog)).

## Open questions

1. **Exact `<project>` path encoding.** Official docs say `<project>` is derived from the git repository, and outside git from the project root, but they do not document the exact directory-name encoding or normalization algorithm for `~/.claude/projects/<project>/memory/` ([memory docs](https://code.claude.com/docs/en/memory#storage-location); [sessions docs](https://code.claude.com/docs/en/sessions#export-a-conversation)).
2. **Formal topic-file schema.** The directory reference shows example topic files with `name`, `description`, and `type: reference`, but the memory page does not define a required frontmatter schema or allowed `type` enum for main-session auto-memory topic files ([directory reference](https://code.claude.com/docs/en/claude-directory); [memory docs](https://code.claude.com/docs/en/memory#storage-location)).
3. **Last-modified timestamp representation.** The changelog states that last-modified timestamps were added to memory files, but the docs do not say whether those timestamps are frontmatter fields, filesystem mtimes, inline Markdown, or another representation ([changelog, v2.1.75](https://code.claude.com/docs/en/changelog)).
4. **Concurrent-write behavior.** Official docs do not specify locking, atomic writes, conflict files, merge behavior, or last-writer-wins semantics for two Claude Code sessions writing the same auto-memory directory ([memory docs](https://code.claude.com/docs/en/memory#runtime-behavior); [changelog, v2.1.63](https://code.claude.com/docs/en/changelog)).
5. **Deduplication and eviction.** Official docs describe Claude moving detail from `MEMORY.md` into topic files and loading only the capped beginning of `MEMORY.md`, but they do not document deduplication, automatic deletion, TTL, or compaction rules for local auto-memory topic files ([memory docs](https://code.claude.com/docs/en/memory#what-gets-loaded)).
6. **Automatic cleanup interaction.** The sessions page documents 30-day cleanup for transcripts, and the directory reference documents project purge deleting auto memory, but no source found clearly says whether automatic transcript cleanup ever removes auto-memory files ([sessions docs](https://code.claude.com/docs/en/sessions#export-a-conversation); [directory reference](https://code.claude.com/docs/en/claude-directory#clean-up-project-state)).
7. **End-of-session memory writes.** The docs state that Claude reads and writes memory during the session and re-injects memory after compaction, but they do not describe a default end-of-session summarization pass that writes memory ([memory docs](https://code.claude.com/docs/en/memory#runtime-behavior); [context-window guide](https://code.claude.com/docs/en/context-window#after-compaction)).
8. **Relationship to API memory internals.** Anthropic documents a Messages API `memory` tool and Managed Agents memory stores, but no official source found says the default Claude Code CLI auto-memory implementation uses those APIs internally ([API memory tool docs](https://platform.claude.com/docs/en/agents-and-tools/tool-use/memory-tool); [Managed Agents memory docs](https://platform.claude.com/docs/en/managed-agents/memory); [memory docs](https://code.claude.com/docs/en/memory#storage-location)).
9. **No official memory export command.** `/export` exports the current conversation, but no fetched official source documents a default memory-specific export or backup command ([commands reference](https://code.claude.com/docs/en/commands); [sessions docs](https://code.claude.com/docs/en/sessions#export-a-conversation)).
