# Agent Memory, Context, and State Engineering

## Memory safety, poisoning, and governance

The attack surface around persistent agent memory is now broad enough that it helps to separate **documented attack classes** from **plausible but still under-documented failure modes**. The best-documented classes in public primary or quasi-primary sources are these:

**Retrieval-path poisoning and poisoned vector stores.** OWASP’s 2025 LLM Top 10 added **LLM08: Vector and Embedding Weaknesses**, explicitly framing embedding generation, storage, retrieval, and ranking as a security boundary. The project says malicious content can be injected into these systems to manipulate outputs or expose sensitive data. Mem0’s 2026 security writeup then points directly to **MINJA** and **AgentPoison** as concrete memory-poisoning families affecting persistent memory stacks. Put differently, “RAG poisoning” is no longer just a content-quality problem. In agent memory, it becomes a durable state corruption problem. (OWASP, 2024-11-18, aging; OWASP GenAI Security Project, accessed 2026-04-23; Mem0 engineering blog, 2026-02-11). citeturn35search11turn35search0turn26search2

**Cross-context leakage in shared memory.** The clearest production documentation I found is in the LangChain Deep Agents production guide, which explicitly warns that **shared memory across users or orgs is a prompt-injection vector** and recommends read-only access for shared policies plus declarative write-denial on shared paths. This is not just a correctness warning. It is a recognition that one principal’s writes can become another principal’s trusted prompt prefix. CrewAI’s memory docs point in the same direction with scoped memory, `source` tags, and `private` visibility controls keyed to the source principal. (LangChain Deep Agents docs, accessed 2026-04-23; CrewAI docs, accessed 2026-04-23). citeturn23search2turn22view0

**MCP tool-descriptor poisoning, including shadowing and rug pulls.** OWASP now treats this as a first-class MCP attack. The OWASP MCP attack page defines **MCP Tool Poisoning** as an indirect prompt-injection attack through tool descriptions or schemas, and the OWASP MCP Top 10 says the category includes **schema poisoning, tool shadowing, and rug pulls**. This matters for memory systems because tool descriptors routinely enter the same trusted context assembly path as memory-derived instructions. (OWASP community attack page, accessed 2026-04-23; OWASP MCP Top 10, accessed 2026-04-23). citeturn35search2turn35search6

**Synthesis-time poisoning.** Public vendor documentation is weaker here than it should be. I did not find a mature, vendor-neutral taxonomy entry naming “nightly synthesis poisoning” as a standard term. But the pattern is real enough that the strongest production defenses I found are architectural: OpenClaw’s dreaming system excludes narrative dream diary artifacts from promotion, allows only **grounded memory snippets** to promote into `MEMORY.md`, and rehydrates snippets from live daily files before writing so stale or deleted evidence is skipped. That is exactly the kind of defense you want against a batch job that might otherwise convert untrusted or deleted content into durable trusted state. (OpenClaw Dreaming docs, accessed 2026-04-23). citeturn10view1turn14view1

**Embedding inversion and embedding leakage.** OWASP clearly treats embeddings as a **sensitive-information disclosure** surface inside LLM08, but I did **not** find a late-2025 or 2026 primary source showing a real production exploit against one of the named memory vendors where embeddings from the memory layer were inverted back into recoverable secrets. The risk is recognized. The deployment-specific evidence is still thin in the public record. (OWASP GenAI Security Project, accessed 2026-04-23). citeturn35search0

On the question of **which production systems have shipped explicit trust models**, the answer is: fewer than the discourse implies.

The most explicit public schema I found is in **OpenClaw’s Memory Wiki**. It treats durable knowledge as structured claims with frontmatter fields including `id`, `text`, `status`, `confidence`, `evidence[]`, and `updatedAt`, while each evidence entry can carry `sourceId`, `path`, `lines`, `weight`, `note`, and `updatedAt`. It also compiles contradiction, low-confidence, stale-page, and open-question dashboards. That is an actual governance surface, not just a vector store with nice marketing. The core `memory-core` plugin still owns recall and promotion, but the wiki layer adds provenance-rich synthesis and lintable belief state. (OpenClaw Memory Wiki docs, accessed 2026-04-23). citeturn10view2turn11view1

**CrewAI** exposes a partial trust model rather than a full one. Records can carry a `source` tag and a `private` flag, memory can be scoped by path, and the system documents how `source`-matched private recall works. It also exposes importance, categories, and metadata inferred by the LLM, plus consolidation actions of `keep`, `update`, `delete`, and `insert_new`. That is meaningful provenance and access control, but it is not the same thing as a candidate-to-trusted promotion state machine with reviewer roles. (CrewAI docs, accessed 2026-04-23). citeturn22view0

**LangChain Deep Agents / LangGraph Store** expose the strongest **ACL model** I found in mainstream orchestration docs, but not a rich confidence/provenance schema. The critical primitives are namespaces, path-level permissions, and read-only enforcement for shared paths. The docs even recommend that organization-wide policies be writable only through application code, not by the agent itself. That is a real least-privilege design, but again, not a full trust-scored memory object model. (LangChain Deep Agents docs, accessed 2026-04-23). citeturn23search2turn23search7

**Mem0** has robust lifecycle operations and metadata plumbing, but not a public “trust model” in the stronger sense. The paper describes a two-phase system where extracted candidate facts are compared to retrieved memories and then mapped by the LLM to `ADD`, `UPDATE`, `DELETE`, or `NOOP`. The docs add `metadata`, categories, immutable memories, explicit delete flows, filtered bulk deletion, event logs, and audit-oriented platform features. What I did **not** find in the public docs is a first-class built-in field set for provenance confidence, promotion status, or reviewer-approved trust tiers. (Mem0 paper, 2025-04-28; Mem0 docs, accessed 2026-04-23; Mem0 OSS prompt config in GitHub, accessed 2026-04-23). citeturn27view0turn24search0turn24search1turn26search19

**Zep / Graphiti** ship a rich **temporal conflict model**, not a broad governance model. The paper shows entity and fact resolution, bi-temporal tracking, `valid_at` and `invalid_at` on edges, and contradiction handling by edge invalidation instead of destructive overwrite. That is strong on supersession and temporal semantics. I did **not** find a public field schema for per-fact confidence, provenance chains, or explicit reviewer promotion states in the paper. (Zep paper, 2025-01-24, aging). citeturn40view0turn39view2

For **Letta, OpenMemory, and vanilla LangGraph stores**, I did not find public primary materials exposing a comparably explicit trust schema of provenance, confidence, and promotion states in the way your chapter likely wants. That absence itself is important.

On **concrete mitigations adopted in production**, the best-evidenced patterns are surprisingly boring and therefore credible: namespace isolation, read-only shared memory, immutable records for high-integrity artifacts, explicit delete APIs, grounded-only promotion, rehydration against live sources before promotion, and source/private attribution on writes. OWASP’s guidance complements those with standard controls around sanitization, isolation, access control, validation of retrieved content, and protection of vector stores and embedding pipelines. (OWASP, 2024-11-18, aging; OpenClaw docs, accessed 2026-04-23; CrewAI docs, accessed 2026-04-23; LangChain Deep Agents docs, accessed 2026-04-23; Mem0 docs, accessed 2026-04-23). citeturn35search11turn10view1turn14view1turn22view0turn23search2turn24search0

For **ACLs, principals, and write attribution in shared or multi-agent memory**, the clearest public answer is that least privilege means **separate proposal from promotion**. LangChain’s file-backed memory model makes shared files read-only and gives per-user or per-subagent writable paths. OpenClaw keeps subagent prompts minimal, and its current context-engine lifecycle only invokes `onSubagentEnded`; `prepareSubagentSpawn` exists in the interface but is not yet called by the runtime. That means the safe default today is narrow subagent scope, explicit write lanes, and a privileged promotion path owned by a parent agent, operator, or batch service. (LangChain docs, accessed 2026-04-23; OpenClaw docs, accessed 2026-04-23). citeturn23search7turn13view0turn13view2

For **PII, secrets, and compliance deletion**, the public materials are more mature on deletion than on tombstoning. Mem0 exposes `delete`, `batch_delete`, and `delete_all` with an important safety change: empty-filter `delete_all` no longer wipes everything, and full-project wipes require explicit `"*"` wildcards. The platform migration docs also claim GDPR support and audit logs. OpenClaw explicitly warns not to store secrets in the workspace, redacts personal and sensitive content before transcript ingestion into dreaming, and rehydrates snippets from live daily files before promotion so deleted material is skipped by later synthesis. That rehydration step is the closest public example I found of a **real tombstone-adjacent implementation** that helps prevent deleted evidence from being silently regenerated into durable memory by a batch promotion job. (Mem0 docs, accessed 2026-04-23; OpenClaw docs, accessed 2026-04-23). citeturn24search0turn24search1turn10view0turn14view1

### Gaps and unknowns

I did **not** find a public, primary-source writeup from Mem0, Letta, Zep, OpenMemory, LangGraph, or CrewAI that fully specifies a production **candidate → trusted → superseded → tombstoned** governance model with principals, reviewer roles, and record-level confidence propagation.

I also did **not** find a public vendor postmortem showing a production **embedding inversion** incident against an agent memory system, even though the risk category is now recognized by OWASP.

And I did **not** find a standard, publicly documented tombstone schema that guarantees downstream synthesis jobs can never regenerate deleted content unless source artifacts are also purged or explicitly masked.

## Evaluation of memory systems

The benchmark landscape is better than it was a year ago, but it is still not testing many of the behaviors senior engineers actually lose sleep over.

**LongMemEval** is the most complete explicit QA benchmark among the public primary sources I reviewed. It defines **five core abilities**: information extraction, multi-session reasoning, knowledge updates, temporal reasoning, and abstention. It operationalizes those through **seven question types**: `single-session-user`, `single-session-assistant`, `single-session-preference`, `multi-session`, `knowledge-update`, `temporal-reasoning`, and `abstention`. The benchmark provides two standard settings, **LongMemEval-S** at roughly **115k tokens** and **LongMemEval-M** at roughly **1.5 million tokens / 500 sessions**. Its QA metric is judge-based accuracy using a prompt-engineered GPT-4o evaluator, and if a system exposes retrieval results the benchmark also reports **Recall@k** and **NDCG@k**. The paper’s own pilot study is a useful reality check: on the shorter `S` setting, GPT-4o dropped from **0.870 oracle** to **0.606** with full-history reading, while ChatGPT memory was far below offline reading even on a much simpler manual setup. (LongMemEval paper, ICLR 2025 / public paper released 2024-2025, aging). citeturn31view1turn32view0

**LoCoMo** is broader but less standardized for later memory-system comparisons. The original benchmark includes **three tasks**: question answering, event summarization, and multimodal dialogue generation. Inside QA, it defines **five reasoning types**: single-hop, multi-hop, temporal, commonsense/world-knowledge, and adversarial. The dataset itself has **50 very long-term dialogues**, around **300 turns** and **9.2k tokens on average**, spread over up to **35 sessions**. The important practical detail is that many later memory-system papers, including Mem0, report only on the **QA subset** and often further narrow the categories they score. That makes cross-paper comparisons much noisier than the benchmark’s reputation suggests. (LoCoMo paper, 2024-02-27, likely stale). citeturn30view0

On **current public frontier scores**, I would treat the field as **unsettled rather than leaderboard-stable**. For LongMemEval-S, Zep’s 2025 paper reports **71.2%** with GPT-4o and **63.8%** with GPT-4o-mini, versus full-context **60.2%** and **55.4%** respectively, while reducing context from **115k tokens** to about **1.6k**. A 2026 preprint, **EverMemOS**, reports an **83.0 overall LongMemEval accuracy** in its search snippet, which is higher than the older published systems I could verify, but I would treat that as **provisional** until you inspect the full paper and protocol yourself. For LoCoMo, a 2026 paper snippet for **Synthius-Mem** claims **94.37% accuracy**, which is higher than the 2025 systems I reviewed, but again I would treat it as a **provisional public claim**, not a reproduced field consensus. (Zep paper, 2025-01-24, aging; EverMemOS preprint, 2026-04-11; Synthius-Mem preprint snippet, 2026-04-16-ish from the search result window). citeturn40view0turn29search9turn28search12

On **what newer benchmarks emerged in 2025 and 2026**, the credible additions are filling different gaps:

**MemoryAgentBench** evaluates memory agents through **four competencies**: accurate retrieval, test-time learning, long-range understanding, and selective forgetting. It is stronger than LongMemEval or LoCoMo on the agentic, incremental-interaction side. (OpenReview / arXiv, 2025-07-09 to 2026 paper cycle). citeturn28search14turn38search11

**MemoryArena** focuses on **interdependent multi-session agentic tasks**, where later success depends on what the agent learns and stores from earlier subtasks. That is materially closer to production than pure post-hoc QA. (Stanford Digital Economy Lab page, 2026-02-18). citeturn28search20turn28search11

**MobileMem** pushes toward heterogeneous, mobile-assistant-style memory workloads and reports that even RAG baselines struggle to clear 50% average accuracy, which strongly suggests LongMemEval-style conversational QA has been comparatively forgiving. (OpenReview PDF, 2026-03, from search snippet). citeturn29search1

**NaturalMem** is aimed at **memory-driven dialogue in realistic personalized settings**, explicitly criticizing explicit probe-style questioning as unnatural for actual personal assistants. That fills a major ecological-validity gap in the current memory literature. (OpenReview PDF, 2025-11-26 / 2026 visibility, aging by your standard only if you date the initial release). citeturn29search13

**LoCoMo-Plus** appears to move beyond raw factual recall toward cue-triggered or beyond-factual cognitive memory behavior, though I would treat it as **emerging** rather than established. (OpenReview PDF, 2026-01). citeturn28search3

**Mem0’s own evaluation** is unusually explicit about what it is and is not measuring. The 2025 paper evaluates on **LoCoMo QA**, not the full LoCoMo benchmark, and specifically excludes the adversarial category because ground-truth answers were unavailable. It reports **F1**, **BLEU-1**, **LLM-as-a-Judge**, plus deployment metrics including **token consumption**, **search latency**, and **total latency**. The headline claims are a **26% relative gain in judge score** over OpenAI’s proprietary memory system baseline, **91% lower p95 latency**, and **more than 90% token savings** versus the full-context method. That is strong deployment-oriented evaluation, but it does **not** measure cross-tenant safety, abstention in the full benchmark sense, long-run self-poisoning, or governance correctness. (Mem0 paper, 2025-04-28). citeturn27view0turn19search0

**Zep’s own evaluation** is more skeptical than many vendor papers. It reports on both **DMR** and **LongMemEval-S**, and explicitly argues that DMR has become too easy because full-context baselines already fit inside modern context windows. On LongMemEval-S, Zep reports **71.2%** with GPT-4o at **2.58s** latency and **1.6k average context tokens**, against full-context **60.2%** at **28.9s** and **115k tokens**. The paper also calls out a meaningful weak spot: **single-session-assistant** performance actually got worse than full-context. That is exactly the kind of contradictory result you want in your handbook. (Zep paper, 2025-01-24, aging). citeturn40view0

I did **not** find a comparable public evaluation package from **Letta** focused on long-term conversational memory benchmarks in the same way. The most public quantitative Letta artifact I reviewed was the **sleep-time compute** paper, which is about offline reasoning and cost/accuracy tradeoffs, not a standard memory benchmark suite. (Letta paper, 2025-04-17). citeturn37search0turn37search1

On your four very specific evaluation gaps:

**Compaction quality.** I did **not** find a credible public benchmark specifically testing whether summarization preserves active state, identifiers, and superseded beliefs while dropping obsolete detail. The closest neighboring literature tests long-context compression or event reconstruction, not production compaction correctness. (LongMemEval paper, aging; MemoryAgentBench paper). citeturn31view1turn38search11

**Passive recall precision/recall.** LongMemEval does at least expose **Recall@k** and **NDCG@k** when retrieval traces are available, which is the best public benchmark support I found for recall-layer evaluation. But it still measures retrieval against curated evidence sessions, not production “entity-triggered passive recall fired at the wrong time” dynamics. (LongMemEval paper, aging). citeturn31view2

**Abstention.** LongMemEval explicitly includes **ABS** and false-premise transformed questions. This is one of the benchmark’s strongest contributions. (LongMemEval paper, aging). citeturn31view3turn32view0

**Self-poisoning over time.** I did **not** find a published benchmark that directly measures whether an agent reinforces and amplifies its own bad priors through repeated recall and re-storage over long horizons. That is a major gap.

On **regression testing patterns**, public production docs are still lighter than they should be. The most concrete public patterns I found are instrumentation surfaces rather than full release-gating recipes: CrewAI emits detailed memory operation events that can be asserted in tests; OpenClaw exposes explainers like `memory promote-explain`, `/dreaming status`, and context inspection commands; LangChain Deep Agents rely heavily on deterministic namespace/path-policy behavior that is amenable to access-control regression tests. What I did **not** find was a mature public “memory regression harness” from the named vendors covering retrieval IDs, compaction goldens, supersession correctness, and deletion non-regeneration in one CI package. (CrewAI docs, accessed 2026-04-23; OpenClaw docs, accessed 2026-04-23; LangChain docs, accessed 2026-04-23). citeturn22view0turn14view1turn13view2turn23search2

### Gaps and unknowns

There is still no broadly adopted benchmark for **compaction correctness**, **passive-recall precision in live runs**, or **self-poisoning over repeated retrieval cycles**.

And there is still no durable public leaderboard discipline. Cross-paper comparisons remain messy because papers choose different backbones, judges, retrieval budgets, and subsets of LoCoMo or LongMemEval.

## Memory lifecycle and temporal validity

If your handbook wants a crisp answer to “which systems implement an explicit lifecycle state machine like candidate, staged, active, pinned, superseded, archived, tombstoned,” the honest answer is: **I did not find a public production system exposing that full state machine as a first-class schema**.

What I found instead were **partial lifecycle implementations**.

**OpenClaw** has the most operationally visible lifecycle, but it is still not the seven-state machine you listed. In dreaming, the system keeps machine state under `memory/.dreams/`, stages “short-term” material during light/REM processing, and promotes only selected grounded candidates during deep sleep into `MEMORY.md`. It additionally distinguishes dream diary artifacts from promotable evidence and provides rollback for grounded historical backfill. That means OpenClaw clearly has at least **staged**, **grounded**, **promoted**, and **rolled-back** notions in practice, even if those are not exposed as a canonical memory-object enum. (OpenClaw Dreaming docs, accessed 2026-04-23). citeturn10view1turn14view1

**CrewAI** exposes a smaller lifecycle at write time through consolidation: when similar content is found above the consolidation threshold, the LLM chooses `keep`, `update`, `delete`, or `insert_new`. This is a real state-transition policy, but again, it is an action policy over records, not a public lifecycle schema. (CrewAI docs, accessed 2026-04-23). citeturn22view0

**Mem0** does something similar with `ADD`, `UPDATE`, `DELETE`, and `NOOP` during its update phase, which is best read as a memory-delta policy rather than a lifecycle model. (Mem0 paper, 2025-04-28; Mem0 prompt config in GitHub, accessed 2026-04-23). citeturn27view0turn26search19

**Graphiti / Zep** do something closer to a proper temporal lifecycle. The paper describes a **bi-temporal model**, with explicit validity windows on facts and contradiction handling through invalidation rather than destructive overwrite. Public HTML rendering obscures some field names, but the temporal extraction prompt clearly defines `valid_at` and `invalid_at`, and the paper states that new contradictory edges can invalidate prior edges while preserving historical records. That is the strongest publicly documented **superseded-but-retained** model I found in a production memory stack. (Zep paper, 2025-01-24, aging). citeturn39view2turn40view0

On **temporal validity fields** specifically, Graphiti is the standout. It uses a reference timestamp per message, supports relative time normalization like “two weeks ago,” and stores temporal information directly on edges. The system distinguishes when a fact became true from when it ceased to be true, and it also models ingestion order separately from real-world validity. That is substantially richer than simple `created_at`. (Zep paper, 2025-01-24, aging). citeturn39view1turn39view2

Outside Graphiti, I did not find comparably explicit public production support for `valid_from`, `valid_until`, and `observed_at` semantics. CrewAI’s LLM analysis can infer dates and metadata on save, but its public docs do not define a built-in temporal validity schema. OpenClaw’s daily notes and dreaming system are inherently time-aware, but they do not surface first-class validity ranges on durable memory entries in public docs. (CrewAI docs, accessed 2026-04-23; OpenClaw docs, accessed 2026-04-23). citeturn22view0turn14view1

On **supersession chains**, the best-documented production policy is Graphiti’s: new contradictory facts do not hard-delete the old edge; they invalidate it by setting its invalidation time based on the new edge, while keeping historical records. That is superior to last-write-wins if you care about explainability and temporal questions. CrewAI and Mem0 both have LLM-mediated update/delete decisions, but the public docs do not expose a comparable persistent supersession chain object. OpenClaw’s Memory Wiki surfaces contradictions and stale pages in dashboards, but it is more of a governance and audit layer than an automatic chain resolver in the core memory store. (Zep paper, 2025-01-24, aging; CrewAI docs, accessed 2026-04-23; Mem0 paper, 2025-04-28; OpenClaw Memory Wiki docs, accessed 2026-04-23). citeturn39view2turn40view0turn22view0turn27view0turn10view2

On **published evidence about memory drift over months of operation**, I did **not** find a real longitudinal field study. LongMemEval and LoCoMo are offline compiled evaluations, not months-long production telemetry. Zep’s paper itself notes that existing benchmarks remain weak and that production scalability and operational realism are under-addressed. In other words, the public literature still knows much more about benchmarked recall under constructed histories than about six-month drift in a live agent serving real users. (LongMemEval paper, aging; LoCoMo paper, likely stale; Zep paper, aging). citeturn31view1turn30view0turn40view0

### Gaps and unknowns

I did not find a public production memory system with a first-class documented state machine matching **candidate, staged, active, pinned, superseded, archived, tombstoned**.

I also did not find a serious longitudinal study measuring drift, self-reinforcement, or stale-belief accumulation over months in deployed agent memory systems.

## Retrieval architecture landscape

On **Anthropic contextual retrieval**, the most important fact is still the original one: Anthropic’s 2024 engineering post reported that **Contextual Retrieval** reduced failed retrievals by **49%**, and by **67% when combined with reranking**. The method consists of **contextual embeddings** plus **contextual BM25**. That is still the main public evaluation number from the builder that created it. I did **not** find a newer official Anthropic benchmark updating those figures in 2025 or 2026, which means the headline result is both useful and, by your standard, **aging**. Anthropic’s 2026 Opus 4.6 release notes do mention stronger long-context retrieval and reasoning, but they do not publish new Contextual Retrieval-specific numbers. (Anthropic engineering post, 2024-09-19, aging; Anthropic release notes, 2026-02-05). citeturn33search0turn33search9turn36search2

On **production adoption and failure modes of contextual retrieval**, I did **not** find public primary-source postmortems from Anthropic or large adopters describing operational failure modes after rollout. The safe takeaway is narrower: if you adopt the method, you are signing up for more indexing and preprocessing work than plain chunk+embed, and you should not assume Anthropic has publicly validated it on agent-memory-specific workloads.

On **RAPTOR**, the core claim remains strong but the source is now **likely stale**. The 2024 paper showed meaningful gains on complex, multi-step document QA and reported a **20-point absolute improvement on QuALITY** when combined with GPT-4. The architecture is conceptually attractive when your corpus is long-form, semantically hierarchical, and relatively stable enough that recursive clustering and summarization amortize well. What I did **not** find is convincing primary-source production evidence from 2025 or 2026 showing RAPTOR as a default retrieval layer for agent memory stacks. So I would treat RAPTOR as a targeted design pattern, not a 2026 default. (RAPTOR paper, 2024-01-31, likely stale). citeturn33search1turn33search4

On **where RAPTOR underperforms**, the public record is thin. I did not find primary production writeups quantifying failure regimes. My best evidence-based inference is that RAPTOR is least attractive where corpora change often, low-latency incremental updates matter, or the retrieval target is tiny factual fragments rather than multi-document synthesis. That inference follows from the architecture’s dependence on recursive summary trees rather than from a public postmortem.

On **GraphRAG**, the official Microsoft research line is quite clear about the use case: it is meant for **query-focused summarization and discovery over large private text corpora**, especially when ordinary snippet retrieval misses the “big picture.” The project’s own notebook says **global search is resource-intensive**, and GitHub issues from the project note that **community report creation can be a large share of total indexing expense** and that incremental indexing is still imperfect. That means the engineering cost is worth paying when your dominant question class is broad synthesis across many documents or communities, not when your workload is mostly exact fact lookup. What I did **not** find from Microsoft is a public **scale threshold** like “only use GraphRAG above X documents or Y tokens.” So I would not invent one. (Microsoft Research blog, 2024-02-13, likely stale; Microsoft paper, 2024-04-24, aging; GraphRAG notebook and issues, 2024-07 to 2024-08, likely stale but still operationally relevant). citeturn34search2turn34search11turn34search22turn34search3turn34search20

On **ColBERT v2 and late-interaction retrieval**, I did not find strong primary evidence from memory-stack builders showing it has become a mainstream default in agent memory by 2026. What I did find was secondary but relevant market signal: a 2026 LightOn technical post argues that late-interaction or multi-vector retrieval is getting renewed attention because of stronger out-of-domain generalization and long-context handling. That suggests it is still worth considering for high-precision retrieval-heavy systems, but public evidence on **agent-memory-specific** cost/benefit versus dense-plus-rerank remains weak. (LightOn blog, 2026-04-09, secondary source). citeturn33search16

On the **reranker landscape**, the public benchmark gap is glaring. Anthropic’s contextual retrieval post is one of the very few primary sources that quantifies improvement from adding reranking at all. Zep’s paper describes support for **RRF**, **MMR**, graph-aware rerankers, and **cross-encoders** as the highest-cost precision layer, and it uses **BGE-m3** models for embedding and reranking in experiments. But I did **not** find a credible public benchmark comparing Voyage, Cohere, Jina, BGE rerankers, and Anthropic-style reranking on **agent-memory workloads** rather than generic web or enterprise RAG. That means any 2026 reranker choice for memory is still mostly a local benchmark exercise. (Anthropic, 2024-09-19, aging; Zep paper, 2025-01-24, aging). citeturn33search0turn40view0

On **hybrid search tuning**, published concrete guidance is sparse. The clearest production default I found is in OpenClaw: hybrid search is enabled with `vectorWeight=0.7`, `textWeight=0.3`, and `candidateMultiplier=4`, while **MMR is off by default** and **temporal decay is off by default**. Those are sensible defaults for agent memory because IDs and config keys still need lexical rescue, but semantic similarity should dominate most personal-memory recall. I did **not** find a broader body of published tuning ranges specifically for agent memory. (OpenClaw docs, accessed 2026-04-23). citeturn17view0turn15view0

### Gaps and unknowns

I did not find a trustworthy public answer to “what corpus size makes GraphRAG worth it” or “what late-interaction retriever is best for agent memory in 2026.”

And I did not find a serious public reranker bakeoff on memory workloads. Most of the industry is still benchmarking retrieval stacks on document QA, not persistent state recall.

## Prospective memory and scheduled or triggered recall

Your suspicion is right: **prospective memory** is a well-established cognitive-science concept, but it has not yet cleanly transferred into the public LLM-agent systems literature as a standardized first-class memory type.

Across the public memory benchmark and system materials I reviewed, the dominant categories remain some variation of **episodic**, **semantic**, **procedural**, and sometimes **working or short-term** memory. LongMemEval covers information extraction, reasoning, updates, and abstention. LoCoMo covers recall, summarization, and multimodal continuation. MemoryAgentBench adds test-time learning and selective forgetting. What I did **not** find was a mature public benchmark or production memory schema centered on “remembering to do X later when trigger Y occurs,” which is the core of prospective memory. (LongMemEval, aging; LoCoMo, likely stale; MemoryAgentBench, 2025-2026). citeturn31view1turn30view0turn38search11

In production systems, future commitments are still mostly handled as **task scheduling**, not as a native memory type. The clearest builder documentation I found is in OpenClaw: its system prompt explicitly tells the agent to use **cron for future follow-up**, reminders, and recurring work rather than sleep loops or polling hacks. That is a very pragmatic design decision. It treats prospective memory as an orchestration and scheduling concern, with memory providing context but not being the scheduler itself. OpenClaw’s dreaming system is scheduled too, but that is consolidation, not user-facing reminders. (OpenClaw system prompt docs, accessed 2026-04-23; OpenClaw Dreaming docs, accessed 2026-04-23). citeturn18view0turn14view1

For **Letta, OpenAI persistent memory, Claude Projects, and related products**, I did **not** find public primary documentation in the materials I reviewed that specifies a native, durable, first-class schema for future commitments with deadlines, triggers, and conditional reminders. That absence matters. It suggests that even where products market “memory,” scheduled commitments are often still implemented in application logic, calendar integrations, task systems, or cron-like orchestration rather than in the memory substrate itself.

On **failure modes**, the literature is still mostly indirect. Benchmarks show that agents struggle with long-range updates, temporal reasoning, and abstention, all of which are prerequisite capabilities for robust prospective memory. But I did **not** find a dedicated published benchmark measuring whether an agent correctly remembers to follow up later, or correctly fails to promise a follow-up when no reliable scheduling mechanism exists. In practice, the classic failure modes remain obvious: commitments stored as plain text in memory but never registered in a scheduler, reminders retrieved too early or too late, and stale commitments resurfacing after conditions changed. The public benchmarking literature simply has not caught up.

### Gaps and unknowns

I did not find a strong public primary-source answer showing that mainstream LLM-agent systems have imported **prospective memory** from cognitive science as a formal object type with trigger semantics.

And I did not find a benchmark that directly measures “remember to do X later” behavior with deadlines, triggers, or condition-dependent reminders.

## Sleep-time compute and dreaming

The **Letta sleep-time compute** paper is one of the few sources in this area that actually gives you quantitative ROI rather than vibes. The central results are clear: sleep-time compute can reduce the amount of **test-time compute needed to reach the same accuracy by about 5x** on **Stateful GSM-Symbolic** and **Stateful AIME**. Scaling sleep-time compute further increases accuracy by up to **13%** on Stateful GSM-Symbolic and **18%** on Stateful AIME. In the multi-query variant, amortizing offline work across related queries reduces **average cost per query by 2.5x**. The paper also says efficacy correlates with **predictability of the future query**, which is exactly the regime question you should put in a production handbook. If the future query distribution is opaque, the ROI story weakens fast. (Letta/UC Berkeley paper, 2025-04-17; Letta blog, 2025-04-21). citeturn37search0turn37search1turn37search3

The catch is that these are **task-benchmark** results, not broad production telemetry from user-facing agent deployments. I did **not** find a second public paper giving equally crisp “nightly synthesis changed next-day user-visible behavior by X% at Y token cost” numbers outside that Letta line of work. There are adjacent systems and writeups, but not much hard evidence.

For **OpenClaw’s current dreaming implementation**, the docs are surprisingly concrete. Dreaming is **opt-in** and **disabled by default**. It has three phases, **Light**, **Deep**, and **REM**, but the docs stress that these are internal cooperative phases, not user-facing modes. Light sleep stages recent short-term material and never writes to `MEMORY.md`. Deep sleep ranks candidates using threshold gates like `minScore`, `minRecallCount`, and `minUniqueQueries`, rehydrates snippets from live daily files before promotion, and is the **only** phase that appends to `MEMORY.md`. REM extracts themes and reflections and also never writes to durable memory. The default managed sweep runs on cron at **`0 3 * * *`** and executes **light -> REM -> deep** in that order. Dreaming writes machine state under `memory/.dreams/`, a narrative diary to `DREAMS.md`, and optional per-phase reports under `memory/dreaming/<phase>/YYYY-MM-DD.md`. (OpenClaw docs, accessed 2026-04-23). citeturn10view1turn14view1

The strongest governance property in OpenClaw’s dreaming design is the one the discourse often ignores: **dream diary text is not a promotion source**. The docs say dream-generated diary and report artifacts are excluded from short-term promotion and that **only grounded memory snippets are eligible to promote into `MEMORY.md`**. There is even a grounded historical backfill lane with reversible staging and rollback for review. That is much stricter than the popular narrative that agents “dream new plans into existence.” (OpenClaw docs, accessed 2026-04-23). citeturn14view1turn14view2

On your sharpest question, the answer is blunt: I did **not** find public evidence that “concrete commitments from last night’s dream” in OpenClaw are reliably executed the next day as an autonomously binding behavior channel. The docs point the other way. The dream diary is for human reading, not a promotion source, and only grounded snippets can become durable memory. So if someone is claiming operational next-day execution from dream prose alone, that claim is ahead of the public documentation. (OpenClaw docs, accessed 2026-04-23). citeturn14view1

### Gaps and unknowns

Beyond Letta’s 2025 paper, I did not find much public measurement of offline consolidation that clearly ties **offline spend** to **next-day behavioral gains** in a way you could directly cost-model into a production stack.

And for OpenClaw specifically, the public docs are excellent on mechanics and governance, but I did not find public outcome metrics showing how often dreaming promotions actually improve downstream task success.

## Cost modeling per stack layer

The best supported cost and latency claims are still coming from memory-system papers rather than from end-to-end production postmortems.

**Hybrid retrieval and compressed memory context** already show very material savings. Mem0’s 2025 paper reports **91% lower p95 latency** and **more than 90% token savings** relative to full-context processing while improving answer quality on LoCoMo QA. Zep’s 2025 paper gives more granular numbers on LongMemEval-S: full-context with GPT-4o used roughly **115k average context tokens** and took **28.9s**, while Zep used around **1.6k average context tokens** and took **2.58s**. GPT-4o-mini showed a similar pattern, **31.3s** down to **3.20s**. Those are not abstract gains. They are strong evidence that once memory retrieval works, “don’t replay the whole transcript” is economically dominant. (Mem0 paper, 2025-04-28; Zep paper, 2025-01-24, aging). citeturn27view0turn40view0

**Prompt caching** is one of the few stack layers with official vendor economics. Anthropic’s prompt caching docs say cache writes cost **1.25x** the base input-token price for the 5-minute cache and cache reads cost **0.1x** the base input-token price, with a default **5-minute TTL** and optional **1-hour TTL**. Anthropic’s release notes also say prompt caching can reduce latency by **up to 80%** and costs by **up to 90%**. The structural rule is straightforward: cache the stable prefix, keep dynamic recall below the cache boundary. OpenClaw’s system-prompt docs mirror that design by explicitly supporting a **stable prefix above the prompt cache boundary** and a **dynamic suffix below it**. (Anthropic docs, accessed 2026-04-23; Anthropic release notes, 2026-02-05; OpenClaw docs, accessed 2026-04-23). citeturn36search0turn36search2turn36search1

That yields a concrete design recommendation for dynamic context assembly with passive recall: put tools, core system instructions, static identity, and other stable context in the cached prefix; inject retrieved memories, fresh evidence, and one-off runtime hints in the suffix. I did **not** find public cache-hit-rate studies from memory-system vendors after adding passive recall, so any claim like “passive recall usually preserves 80% cache hit rate” would be made up. What the public record supports is the **assembly pattern**, not the hit-rate number.

For **GraphRAG**, the public cost story is mostly qualitative but still useful. Microsoft’s project materials and GitHub issues repeatedly characterize **global search** as **resource-intensive** and community-report creation as a major share of indexing expense. So the engineering-cost warning is real, even if the project does not publish a neat dollars-per-million-tokens breakdown. Similarly, I did not find a production cost ledger for **RAPTOR** preprocessing, though the architecture obviously adds offline summarization and reindexing costs. (Microsoft GraphRAG materials, 2024, aging to likely stale operationally). citeturn34search22turn34search3turn34search20

For **reranking**, the public economics are weak. Zep’s paper states the obvious but important truth that **cross-encoder reranking** gives the highest-quality precision layer at the **highest computational cost**. Anthropic’s contextual retrieval post shows that adding reranking on top of contextual retrieval materially improves retrieval quality, but it does not price the reranker separately. So a precise “reranker tax per request” still needs local benchmarking. (Zep paper, 2025-01-24, aging; Anthropic, 2024-09-19, aging). citeturn40view0turn33search0

For **nightly synthesis or dreaming**, hard public cost numbers are almost absent. Letta’s sleep-time compute work gives test-time and per-query efficiency in benchmark settings, and OpenClaw documents the scheduler and promotion mechanics, but I did not find a public workup that decomposes nightly consolidation into an operational token bill per user per day across a live deployment. That is still a live gap in the literature.

On whether **Mem0, Letta, and Zep** publish scale cost breakdowns: **Mem0** and **Zep** do publish meaningful token and latency comparisons in benchmark settings; **Letta** publishes sleep-time compute efficiency on benchmark and case-study tasks; none of the three, in the public materials I reviewed, publish a mature “at X DAU, with Y average memories per user, the monthly retrieval + consolidation + caching bill is Z” operating model. (Mem0 paper, 2025-04-28; Zep paper, 2025-01-24, aging; Letta paper, 2025-04-17). citeturn27view0turn40view0turn37search0

### Gaps and unknowns

I did not find public, primary-source **cache-hit-rate data** for passive recall under Anthropic prompt caching.

I also did not find clean per-layer dollarized cost breakdowns for GraphRAG, RAPTOR, or nightly synthesis in production agent memory systems.

## What surprised me

The biggest surprise is how **little explicit trust modeling** has actually shipped in public. The loudest memory vendors talk a lot about persistence and recall, but the strongest public trust/provenance schema I found was not from the most hyped memory layer. It was from **OpenClaw’s Memory Wiki** and from the rather prosaic **ACL/path-policy model in LangChain Deep Agents**. The industry conversation is still ahead of the shipped governance surface. (OpenClaw docs, accessed 2026-04-23; LangChain docs, accessed 2026-04-23). citeturn10view2turn11view1turn23search2turn23search7

The second surprise is that **easy benchmarks are still distorting the discourse**. Zep’s own paper effectively says DMR is no longer a meaningful differentiator because full-context baselines already fit the conversation. That is a rare case of a vendor paper honestly explaining why an older benchmark is too weak. If your handbook has a chapter on evaluation, I would elevate that point. Many “memory wins” are still benchmark-selection wins. (Zep paper, 2025-01-24, aging). citeturn40view0

The third surprise is that the most serious current benchmark, **LongMemEval**, still leaves out exactly the production concerns practitioners care about most: compaction fidelity, passive recall precision in live runs, self-poisoning, and durable future commitments. It is much better than the earlier generation, but it is still mainly a curated QA benchmark with retrieval hooks, not a comprehensive agent-memory reliability suite. (LongMemEval paper, aging). citeturn31view1turn31view2

The fourth surprise is that **“dreaming” systems are more conservative than the discourse around them**. OpenClaw’s current implementation does **not** let diary prose mint durable memory. It enforces grounded evidence, threshold gates, live-file rehydration, and clear separation between human-readable dream output and promotable records. In other words, the production-minded implementation is much less mystical than the discourse. That should change how a senior engineer designs offline consolidation: treat it as a governed evidence pipeline, not a free-form reflection loop. (OpenClaw docs, accessed 2026-04-23). citeturn14view1

The fifth surprise is that **least privilege for agent memory today is mostly filesystem and namespace engineering**, not fancy model-native trust reasoning. Read-only shared memory, per-user namespaces, per-subagent writable scopes, explicit promotion lanes, and delete-before-regenerate safeguards are doing much more real safety work than confidence scores alone. That is good news for production engineers because it means many of the highest-value controls are boring, testable, and available now. (LangChain docs, accessed 2026-04-23; OpenClaw docs, accessed 2026-04-23; CrewAI docs, accessed 2026-04-23). citeturn23search2turn23search7turn13view0turn22view0
