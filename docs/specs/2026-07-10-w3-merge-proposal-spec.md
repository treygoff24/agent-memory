Plan: `docs/plans/2026-07-10-memora-lessons-memorum-upgrades.md` §W3 (r5, approved). ## 1. The operation

A **merge proposal** consolidates N near-duplicate memories (N ≥ 1; N = 1 is a rewrite-in-place proposal) into one replacement memory. Proposals are **generated by a dream pass, approved by a human (or explicit auto-approve policy later — none this arc), and applied by the daemon**. Nothing auto-merges.

### 1.1 Proposal record

`MergeProposal { proposal_id (ULID), source_ids: Vec<MemoryId> (≥1), replacement: { title, summary, body, abstraction, cues, type, scope, namespace fields }, provenance: union of source entities/evidence/refs unless overridden, similarity_evidence: pairwise cosines + lane, created_by: dream run id, created_at, status: Proposed | Approved | Rejected | Invalidated | Applying | Applied | RolledBack }`

Replacement classification is **not stored as trusted input**: at apply time the replacement payload runs the full W2 §A4 pipeline (normalize/cap → classify combined payload → decision → validate → persist); the floor is the **strictest source classification** — the pipeline may only tighten, never loosen. `secret` refuses the apply and invalidates the proposal.

### 1.2 Storage — device-local runtime, not the index, not synced

Proposals and the apply journal live under `<runtime>/governance/merge-proposals/<proposal_id>.json` (+ `journal.jsonl` per proposal), fsync-on-write, single-writer daemon. Rationale: no index-schema contention (schema 6 stays W2-only, v4 P2 keeps 6→7), nothing syncs (Stream I posture: proposals are device-local governance intent; the *result* syncs as ordinary canonical supersedes), and a lost proposal is cheap (the next dream re-proposes). The apply journal is the crash-recovery source of truth and is retained after terminal states for audit.

## 2. Candidate fences (normative — all must hold at proposal time AND re-validate at apply)

- status ∈ {active, pinned}; **pinned sources additionally require the human approval to name the pinned id explicitly** (no silent merge of pinned)
- not pending review; not candidate/quarantined; `review_state != pending`
- not encrypted-tier; passive-recall-eligible
- same scope, same canonical namespace, same memory `type`
- sensitivity-compatible: no-downgrade rule — the replacement's classification floor is the strictest source; sources whose sensitivities imply different serving surfaces (e.g. `public` + `confidential`) are NOT compatible
- not in W1's import-repair lineage sets, not in the W5 backfill-manifest exclusion set
- pairwise cosine over **abstraction vectors** ≥ threshold (start 0.8, tuned on the W0 dev split only); live NN-distance histogram reported before any live proposals

## 3. Apply algorithm — journaled, idempotent, resumable (never "atomic")

The substrate cannot atomically write N+1 canonical files + git + index (+ vector store, per §10.2.1 precedent). The apply is a journaled sequence with a provable serving invariant instead.

**Serving invariant (holds at every crash point):** passive recall, `memoryd search`, and the review queue never surface the replacement and any source simultaneously as servable memories. Formally: replacement is non-servable ∨ all sources are superseded.

Phases, each journaled (fsync) **before** its substrate action:

1. **`validated`** — approval-time pre-flight: every fence in §2 re-checked against current state; per-source `expected_base_hash` captured. Any failure → status `Invalidated`, terminal, whole proposal (never partial).
2. **`staged`** — replacement written into the canonical tree as **status `candidate`, `trust_level: candidate`, `requires_user_confirmation: false`, `write_policy.policy_applied: "merge-staged-v1"`, `supersedes: [all source ids]`**, via the W2 §A4 pipeline. Non-servability is enforced on THREE surfaces and tested: excluded from passive recall and `memoryd search` (candidate status + the `merge-staged-v1` marker), and excluded from the review queue (the marker overrides the candidate-review trigger). The shipped supersession primitive writes replacements active-first — staging is deliberately NOT that primitive.
3. **`superseding(i)`** — one journal row per source, then mutate source i: `status: superseded`, `superseded_by: [replacement_id]`, guarded by its captured `expected_base_hash`. A hash-guard failure here (concurrent human edit) → **rollback** (below). Emits the existing per-source supersession event.
4. **`activating`** — flip replacement `candidate → active` (trust per lifecycle matrix), remove the staging marker. Emits terminal `MergeApplied` event (additive event kind, §12.2 amendment).
5. **`done`** — proposal status `Applied`.

**Crash recovery (startup + dream-entry reconcile, before reads are served):** an `Applying` journal is resumed **forward** from its last completed phase — re-validation is NOT repeated mid-sequence (the base-hash guards on each remaining step carry the safety); each step is idempotent (already-superseded source with matching `superseded_by` = no-op; already-active replacement = no-op).

**Rollback (validation failure mid-apply, phase 3 only):** journal `rolling_back`, then: re-activate every already-superseded source (restore `status: active`, remove `superseded_by: replacement_id`), tombstone the staged replacement (reason `merge-rollback`), status `RolledBack`, terminal. Rollback steps are themselves journaled and idempotent under the same recovery rule.

**Idempotent replay:** re-applying an `Applied` proposal is a complete no-op (every step no-ops). Two-clone behavior: proposals never sync, so cross-device replay cannot happen; the applied result syncs as ordinary supersedes and merges under existing §14 rules.

## 4. Decision: no multi-supersede substrate primitive this arc

The plan requires this spec to decide the alternative (one git commit + one index txn primitive, budgeted as Stream A work). **Decision: not this arc.** The journaled sequence meets the serving invariant without substrate surgery, and a "true" atomic primitive would still be partial — the vector store is outside any SQLite/git transaction (§10.2.1), so atomicity claims would be exactly the kind the round-2 plan review killed. Revisit only if live behavior shows tail risks the journal can't cover; the budget note stands in the plan.

## 5. Surfaces

- `memoryd review merges` (list) / `review merges approve <id>` / `review merges reject <id>` — v1 agent envelope, additive; approve triggers the apply inline (daemon call) and reports the terminal phase.
- Doctor: counts of proposals by status; a stuck `Applying` journal older than one reconcile cycle is a doctor error.
- Dream integration: the merge-candidate job runs in the existing dream machinery after `abstraction_compile`; per-run proposal cap (default 10) to bound review load.

## 6. Gate mapping (plan §W3)

Seeded-corpus proposes right merges only; every §2 fence has a violating fixture (incl. privacy-incompat and pinned-without-named-approval); **crash-injection on the apply path**: kill between phases 2/3, 3/3-i, 3/4, 4/5 → reconcile completes forward; hash-guard failure mid-3 → rollback completes; serving-invariant assertion at every phase boundary; idempotent re-apply; two-clone convergence of the synced result.
