## 1. The operation

A **merge proposal** consolidates N near-duplicate memories (N ≥ 1) into one replacement memory. **N = 1 is NOT rewrite-in-place**: it uses the identical new-ID supersession algorithm as N > 1 (a same-ID replacement would violate self-reference validation and fork the algorithm for no benefit). Proposals are **generated by a dream pass, approved by a human (or explicit auto-approve policy later — none this arc), and applied by the daemon**. Nothing auto-merges.

### 1.1 Proposal record

`MergeProposal { proposal_id (ULID), source_ids: Vec<MemoryId> (≥1), replacement: { title, summary, body, abstraction, cues, type, scope, namespace fields }, provenance: union of source entities/evidence/refs unless overridden, similarity_evidence: pairwise cosines + lane, created_by: dream run id, created_at, status: Proposed | Approved | Rejected | Invalidated | Applying | Applied | RolledBack | Quarantined }`

`Quarantined` is terminal-until-operator: a proposal whose apply or rollback hit a state it must not resolve automatically (§3.4, §3.5). Doctor reports it as an error.

Replacement classification is **not stored as trusted input**: at apply time the replacement payload runs the full W2 §A4 pipeline (normalize/cap → classify → decision → validate → persist). **Merge staging is a registered W2 §A4 generation context** (alongside dream compile and backfill): the dual-classification + outcome-rebind rule applies — if the combined payload classifies stricter than the body-only payload, `abstraction`/`cues` are dropped and the write rebinds to the body-only outcome. **Floor composition order: rebind first, then floor** — the persisted outcome is `stricter(rebound outcome, strictest source classification)`; the floor may only tighten the rebound outcome, never loosen it, and dropping generated fields never lets the replacement fall below the floor. `secret` in either classification refuses the apply and invalidates the proposal.

### 1.2 Storage — device-local runtime, not the index, not synced

Proposals and the apply journal live under `<runtime>/governance/merge-proposals/<proposal_id>.json` (+ `journal.jsonl` per proposal), fsync-on-write, single-writer daemon. Rationale: no index-schema contention (schema 6 stays W2-only, v4 P2 keeps 6→7), nothing syncs (Stream I posture: proposals are device-local governance intent; the *result* syncs as ordinary canonical supersedes), and a lost proposal is cheap (the next dream re-proposes). The apply journal is the crash-recovery source of truth and is retained after terminal states for audit.

### 1.3 Journal format — framed, checksummed, fail-closed

`journal.jsonl` records are single-line JSON objects `{ seq, proposal_id, record: <phase payload>, record_sha256 }` where `record_sha256` = sha256 of the canonical (serialized) `record` value. Rules:

- Each phase writes an **intent record before its substrate action** and a **completion record after**. Because every action is idempotent and CAS-guarded (§3.3), an intent with no completion is safely re-driven on resume.
- **Tail rule (exhaustive):** an **unparseable or incomplete final line** is a torn write — treated as "record never written," truncate and resume from the last complete record. A **parseable record whose `record_sha256` fails verification — anywhere, including the final record** — is corruption: fail closed, proposal `Quarantined`. An **unparseable non-final line** is corruption: `Quarantined`. Never guess forward past corruption. *(Recorded deviation from triage row 8's blanket "torn tail ⇒ quarantine" wording: a torn final line is the expected artifact of every crash-mid-write — quarantining it would quarantine every crash-injection test and defeat forward-resume. Checksum-verified corruption anywhere still quarantines.)*
- **Quarantine consequence:** no further apply or rollback steps, doctor error, operator repair.
- **Reconcile ordering:** merge-journal reconcile runs inside daemon startup reconciliation **before any read surface is served**, and again at dream entry. A stuck `Applying` journal older than one reconcile cycle is a doctor error (§5).

## 2. Candidate fences (normative — all must hold at proposal time AND re-validate at apply)

- status ∈ {active, pinned}; **pinned sources additionally require the human approval to name each pinned id explicitly** via the approval DTO (§5) — no silent merge of pinned
- not review-pending per the **shipped review-membership predicate** (the same predicate `memoryd review` uses, which recognizes all pending spellings — `pending`, `pending_review`, `pending-review`); `requires_user_confirmation != true`; not candidate/quarantined
- not encrypted-tier; passive-recall-eligible
- **not a source in any other non-terminal merge proposal** (proposal-level source exclusivity, checked against the proposal store at validation)
- **no Stream I claim lock** on any source (claim-locked sources are rejected at validation)
- same scope, same canonical namespace, same memory `type`
- sensitivity-compatible: no-downgrade rule — the replacement's classification floor is the strictest source; sources whose sensitivities imply different serving surfaces (e.g. `public` + `confidential`) are NOT compatible
- not in W1's import-repair lineage sets, not in the W5 backfill-manifest exclusion set
- pairwise cosine over **abstraction vectors** ≥ threshold (start 0.8, tuned on the W0 dev split only); live NN-distance histogram reported before any live proposals

## 3. Apply algorithm — journaled, idempotent, resumable (never "atomic")

The substrate cannot atomically write N+1 canonical files + git + index (+ vector store, per §10.2.1 precedent). The apply is a journaled sequence with a provable serving invariant instead.

### 3.1 Non-servability predicate (central, single definition)

A memory is **merge-non-servable** iff `status == candidate ∧ write_policy.policy_applied == "merge-staged-v1"`, or `status == superseded`. This predicate is implemented ONCE and applied by **every** read lane: passive recall, hybrid (fused) search, **FTS-only degraded search** (shipped `query_chunks` filters only `metadata_only`/`passive_recall` — the W3 build must add the status/marker filter there), **delta recall**, and startup recall. Gate includes forced-FTS-degraded and delta-recall tests asserting a staged candidate never surfaces.

**Direct `memoryd get` by id is deliberately NOT gated** by this predicate: direct get is the inspection surface the review CLI itself needs. A staged replacement returned by `get` carries the `merge-staged-v1` marker in its envelope so callers can tell.

**Serving invariant (holds at every crash point):** no gated read lane surfaces the replacement and any source simultaneously as servable memories. Formally: replacement is merge-non-servable ∨ all sources are superseded.

### 3.2 Availability-gap decision (ratified)

During phase 3→4 there is a window where sources are already superseded but the replacement is not yet active — for the merged fact, **neither copy is servable**. Decision: **the gap is accepted.** It is single-device, bounded to consecutive local fs/index operations with no awaits or network between them, and a crash inside the gap is covered by forward-resume (reconcile completes activation before reads are served). The gate asserts post-crash availability: after kill-between-phases-3-and-4 + restart, the replacement is active and servable.

### 3.3 Phases

Each phase writes its journal intent (fsync) **before** its substrate action and a completion record after.

1. **`validated`** — approval-time pre-flight: every fence in §2 re-checked against current state; per-source captured tuple `(expected_base_hash, original_status, original_trust_level)` journaled — rollback restores exactly this tuple. Any failure → status `Invalidated`, terminal, whole proposal (never partial).
2. **`staged`** — replacement written into the canonical tree as **status `candidate`, `trust_level: candidate`, `requires_user_confirmation: false`, `write_policy.policy_applied: "merge-staged-v1"`, `supersedes: [all source ids]`**, via the W2 §A4 pipeline **with aux enqueue suppressed** (§3.6). Non-servable everywhere per §3.1. The shipped supersession primitive writes replacements active-first — staging is deliberately NOT that primitive.
3. **`superseding(i)`** — one journal row per source, then mutate source i as a **CAS**: succeeds only if the source still matches `(expected_base_hash, status == original_status, superseded_by empty)`. On success: `status: superseded`, `superseded_by: [replacement_id]`; **a pinned source is simultaneously demoted `trust_level: pinned → trusted`** (shipped lifecycle forbids `superseded` + `pinned` trust, `validate.rs`; the original tuple lives in the journal for rollback). Non-pinned sources keep their trust. A CAS failure here (concurrent human edit) → **rollback** (§3.5). Re-drive no-op rule: a source already `(superseded, superseded_by == [replacement_id])` is complete — any other already-superseded shape is a CAS failure.
4. **`activating`** — flip replacement `candidate → active` as a CAS on the staged state; remove the staging marker; **explicitly write the replacement's `trust_level` = `untrusted` if any source's original trust was `untrusted`, else `trusted`** (grounded in the shipped enum: `active` permits only `trusted`/`untrusted`; `pinned` never propagates, so all-pinned sources also yield `trusted`). Activation is an **entering-servable transition** (W2 §A3 matrix row, package r4): upsert current-hash abstraction/cue rows and enqueue aux jobs under the stale-hash fence (§3.6). Emit the terminal `merge_applied` event (§3.7).
5. **`done`** — proposal status `Applied`.

**Lifecycle-transition amendments (required by this spec; shipped `validate.rs` rejects these today):** `candidate → active` is legal iff the memory carries the `merge-staged-v1` marker (merge activation); `pinned → superseded` is legal only inside a merge apply whose approval named that pinned id, **and carries the mandatory trust demotion `pinned → trusted`** (so the resulting tuple is lifecycle-valid); `superseded → active` **and `superseded → pinned`** (rollback restore) are legal only inside a journaled merge rollback, and restore the captured original `(status, trust_level)` tuple exactly — including `(pinned, pinned)`. These are additive matrix rows, not a general loosening.

**Crash recovery (startup + dream-entry reconcile, before reads are served):** an `Applying` journal is resumed **forward** from its last completed record — re-validation is NOT repeated mid-sequence (the CAS guards on each remaining step carry the safety); each step is idempotent under the no-op rules above.

**Concurrency:** the daemon holds a **single merge-apply mutex** — at most one proposal applies (or rolls back) at a time; the dream scheduler is already serialized with applies. Combined with proposal-level source exclusivity (§2) and per-step CAS, no two proposals can race a source, and human edits race-lose safely (CAS failure → rollback, never overwrite).

### 3.4 Quarantine (fail-closed terminal)

A proposal is `Quarantined` when: journal corruption (§1.3), a rollback CAS failure (§3.5), or any state the algorithm must not resolve automatically. Quarantine takes no further substrate actions, is a doctor error, and waits for operator repair (manual `review merges reject` after inspection, or hand-repair of the affected memories).

### 3.5 Rollback (CAS failure in phase 3 only)

Journal `rolling_back`, then for each already-superseded source, a **CAS restore**: succeeds only if the source is still `(superseded, superseded_by == [replacement_id])` — proposal-owned. On success restore the captured original `(status, trust_level)` tuple (including `(pinned, pinned)`) and clear `superseded_by`. **Restore is an entering-servable transition** (W2 §A3 matrix row, package r4): the restored source's current-hash abstraction/cue rows are upserted and aux jobs enqueued — rollback must return the source to full aux retrievability, not just canonical status. **A rollback CAS failure (the superseded source was edited or re-owned after supersession) → stop immediately, proposal `Quarantined`** — never overwrite the newer state. Then tombstone the staged replacement (reason `merge-rollback`), which also purges any derived aux state (§3.6), status `RolledBack`, terminal. Rollback steps are themselves journaled and idempotent under the same recovery rule.

### 3.6 Aux embedding state (W2 §A3 interaction)

Staging (phase 2) creates **no** abstraction/cue aux rows, no aux pending jobs, no aux vectors — the §A4 pipeline runs with aux enqueue suppressed for `merge-staged-v1` writes. Aux state is materialized exactly at the **entering-servable transitions** (W2 §A3 matrix row, package r4): replacement activation (phase 4) and rollback restore (§3.5) both upsert current-hash abstraction/cue rows and enqueue aux jobs under the stale-hash fence — the W2 Enqueue operation, not a bare job insert. Rollback's tombstone of the staged replacement purges any aux state that may exist (defense-in-depth; there should be none). Query-side: aux lanes inherit the §3.1 predicate — a staged candidate has no aux rows to serve, and superseded sources' aux rows are deleted per the W2 §A3 lifecycle matrix when their status flips.

### 3.7 Events (defined here — these do not exist in shipped code)

Shipped `events/log.rs` has no per-source supersession event (`Superseded` is explicitly deferred) and no merge event; this spec **adds two additive event kinds** rather than citing phantom ones:

- **`merge_applied`** (terminal, emitted once at phase 4 completion): `{ proposal_id, replacement_id, source_ids, per_source: [{ id, base_hash, original_status }], created_by_dream_run }`. Idempotency key = `proposal_id`; replay of an `Applied` proposal emits nothing (emission is journaled as part of phase 4's completion record). Event-mirror mapping: mirrored like other governance events; consumers treat `proposal_id` as the dedup key.
- Individual file mutations continue to emit the existing `WriteCommitted` events; no per-source `Superseded` event is introduced this arc (the per-source detail rides inside `merge_applied`).
- Rollback emits **`merge_rolled_back`** `{ proposal_id, replacement_id, restored_source_ids }`, same idempotency rule.

**Idempotent replay:** re-applying an `Applied` proposal is a complete no-op (every step no-ops, no events re-emitted). **Two-clone contract:** proposals and journals never sync (device-local runtime); only the resulting canonical supersedes/new memory sync, and converge under existing §14 merge rules — the gate's two-clone test asserts canonical-content equality (spec §13.6.1) after device A applies and both sync, plus replay-idempotence keyed by `proposal_id` on device A. Cross-device replay cannot happen by construction; the plan's "idempotent replay across two clones" is satisfied by this contract (same-device replay + synced-result convergence), not by syncing proposals.

## 4. Decision: no multi-supersede substrate primitive this arc

The plan requires this spec to decide the alternative (one git commit + one index txn primitive, budgeted as Stream A work). **Decision: not this arc.** The journaled sequence meets the serving invariant without substrate surgery, and a "true" atomic primitive would still be partial — the vector store is outside any SQLite/git transaction (§10.2.1), so atomicity claims would be exactly the kind the round-2 plan review killed. Revisit only if live behavior shows tail risks the journal can't cover; the budget note stands in the plan.

## 5. Surfaces

- `memoryd review merges` (list) / `review merges approve <id> [--approve-pinned <memory_id>]...` / `review merges reject <id>` — v1 agent envelope, additive; approve triggers the apply inline (daemon call) and reports the terminal phase. **`--approve-pinned` is required once per pinned source id; approval fails closed listing unacknowledged pinned ids.** Reject on a `Quarantined` proposal is the operator-repair exit.
- Doctor: counts of proposals by status; a stuck `Applying` journal older than one reconcile cycle is a doctor error; any `Quarantined` proposal is a doctor error.
- Dream integration: the merge-candidate job runs in the existing dream machinery after `abstraction_compile`; per-run proposal cap (default 10) to bound review load.

## 6. Gate mapping (plan §W3)

Seeded-corpus proposes right merges only; every §2 fence has a violating fixture (incl. privacy-incompat, pinned-without-named-approval, source-in-another-proposal, claim-locked source); **crash-injection on the apply path**: kill between phases 2/3, 3/3-i, 3/4, 4/5 → reconcile completes forward; CAS failure mid-3 → rollback completes; rollback CAS failure → quarantine (no overwrite); journal fixtures (torn final line resumes; checksum-invalid final record quarantines; corrupt interior quarantines); serving-invariant assertion at every phase boundary **on every gated lane incl. forced-FTS-degraded and delta recall**; availability restored after kill-in-gap + restart; idempotent re-apply emits no duplicate events; two-clone convergence of the synced result; lifecycle-matrix fixtures for the amended transitions (legal inside merge context, still illegal outside), **including the full pinned path: pinned supersede demotes trust, all-pinned activation yields `trusted`, rollback restores `(pinned, pinned)`**; rollback-retrieval fixture: a rolled-back source is servable through abstraction-vector retrieval again (aux rows + jobs re-materialized); merge-staging classification fixtures: dual-classify/rebind fires for the replacement and the persisted outcome never falls below the strictest-source floor.
